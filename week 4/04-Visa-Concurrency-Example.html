Real-World Concurrency

How Visa processes 60,000 transactions per second without losing a cent

The Setup: Black Friday

Visa's Problem

60,000
transactions per second on peak days

Every swipe, tap, and online checkout is a transaction: "Charge $47.99 to card ending 4521 at Target." Each one needs to read the balance, check the credit limit, approve or decline, and write the new balance.

Why serial execution is impossible

Each transaction takes roughly 10ms (disk reads, credit check, write).

Running them one at a time: 60,000 × 10ms = 600 seconds to process 1 second of transactions.

You fall 599 seconds behind every second. The system collapses in minutes. Millions of customers stare at "Transaction pending..." on Black Friday.

Course concept → Serial schedules are always correct, but catastrophically slow for real workloads. This is why databases MUST interleave operations from different transactions.

The Danger: What Happens Without Concurrency Control

Your Visa card has a $500 credit limit and $400 available balance. Two charges hit at the same time:

T1: Nike.com — $80 shoes

r1(bal), w1(bal)
Reads balance, subtracts $80, writes new balance.

T2: Spotify — $15 auto-renewal

r2(bal), w2(bal)
Reads balance, subtracts $15, writes new balance.

The bad interleaving (no protection)

WR Conflict → Anomaly: Lost Update

Time T1: Nike $80 T2: Spotify $15 Balance on disk
1 r1(bal) → reads $400 $400
2 r2(bal) → reads $400 $400
3 computes $400 − $80 = $320 computes $400 − $15 = $385 $400
4 w1(bal) → writes $320 $320
5 w2(bal) → writes $385 $385

Final balance: $385. The $80 Nike purchase just vanished. You got free shoes. Visa loses $80.

Why did this happen?

T2 read the balance before T1's write took effect. It used the stale value ($400) instead of the updated one ($320). Then T2's write overwrote T1's result.

This is a WR conflict that became a real anomaly — T1 wrote, T2 read the old value, and the final state matches no serial ordering.

Course concept → Conflict: w1(bal) and r2(bal) are from different transactions, on the same data item, and one is a write. That's a WR conflict. In this bad interleaving, the conflict produced an anomaly (lost update).

The scale of the damage

Without protection

At 60,000 TPS, even 0.1% anomaly rate = 60 lost transactions/second. At an average of $50 each, that's $3,000/second in errors — roughly $260 million per day.

With concurrency control

The DBMS detects conflicts, prevents anomalies, and guarantees every interleaved schedule is equivalent to some serial execution. $0 lost.

The Solution: How Visa Prevents Anomalies

Visa's database applies the exact concepts from the course. Here is how each piece fits together:

1 Each card swipe is a transaction

Every purchase is wrapped in a transaction with atomicity guarantees:

BEGIN TRANSACTION
    read(balance)                   -- check available credit
    new_bal = balance - purchase_amt
    IF new_bal < 0 → ABORT          -- decline the card
    write(balance = new_bal)         -- deduct the charge
COMMIT
Course concept → Transaction: A logical unit of work that either fully completes or fully rolls back. If the system crashes between the read and write, WAL (Write-Ahead Logging) ensures the partial change is undone. Atomicity + Durability from Week 4.

2 The DBMS interleaves but detects conflicts

The database runs thousands of transactions simultaneously. But when two transactions touch the same card (same data item), the system identifies the conflict:

  • T1 writes balance, then T2 reads balance → WR conflict on the same card
  • The conflict graph gets an edge: T1 → T2 (T1 must logically precede T2)

As long as the conflict graph stays acyclic, the interleaving is safe and the DBMS allows it.

Course concept → Conflict graph: Nodes = transactions, directed edges = conflict orderings. Acyclic graph = conflict serializable = safe to execute.

3 If a cycle forms, abort a victim

If both T1 and T2 read the old balance and both try to write (creating edges T1 → T2 and T2 → T1), the conflict graph has a cycle.

Cycle detected: T1 ↔ T2

T1 → T2: T1's write must happen before T2's read.

T2 → T1: T2's write must happen before T1's read.

Both can't be first. Contradiction.

The DBMS picks a victim — typically the transaction that has done the least work — aborts it, rolls back its changes using the WAL undo log, and retries it.

In practice: your Spotify charge gets declined for a split second, then automatically retries 50ms later. You never notice.

Course concept → Victim selection: When the conflict graph has a cycle, the DBMS must abort one transaction to break it. Strategies include: youngest transaction, least work done, or lowest priority.

4 The correct interleaving

After concurrency control, the schedule looks like this:

Conflict serializable schedule — equivalent to serial T1 then T2

Time T1: Nike $80 T2: Spotify $15 Balance
1 r1(bal) → $400 (blocked — conflict on same card) $400
2 w1(bal) → $320 (still waiting) $320
3 COMMIT ✓ r2(bal) → $320 $320
4 w2(bal) → $305 $305
5 COMMIT ✓ $305 ✓

Final balance: $305. Both charges went through. $400 − $80 − $15 = $305. Correct.

Course concept → Serializable schedule: This interleaved schedule produces the same result as the serial order T1 then T2. The conflict graph has only one edge (T1 → T2), no cycle, so it's conflict serializable. Topological sort gives the equivalent serial order: T1, T2.

Why This Scales to 60,000 TPS

Most of those 60,000 transactions are on different cards. Your Nike purchase and someone else's Starbucks coffee don't conflict at all — different data items, no conflicts, no edges in the graph. They run fully in parallel with zero coordination.

Conflicts only happen when two transactions hit the same card at the same time. With millions of active cards, this is rare. So the system gets:

Massive parallelism

99.9% of transactions touch different cards and run with zero waiting. Full speed.

Guaranteed correctness

The 0.1% that do conflict are serialized (one waits for the other) or aborted and retried. No anomalies ever.

The numbers:

60,000 TPS across ~500 million active Visa cards.

Probability of two transactions hitting the same card in the same 10ms window: extremely low.

So the conflict graph at any moment has 60,000 nodes but almost no edges. Trivially acyclic. Almost everything runs in parallel.

The Takeaway

Concurrency control is not an abstract academic exercise. It is the reason your credit card works reliably when 60,000 other people are swiping theirs at the exact same moment.

  • ✓ Transactions wrap each swipe into an atomic unit
  • ✓ Interleaving gives the throughput needed for 60,000 TPS
  • ✓ Conflict detection identifies the rare cases where two charges hit the same card
  • ✓ Conflict graphs check if the interleaving is safe (acyclic) or dangerous (cyclic)
  • ✓ Victim selection + abort + retry breaks any cycles, with delays so small you never notice
  • ✓ The result: maximum parallelism with guaranteed correctness — no free shoes, no lost charges, no inconsistent balances
