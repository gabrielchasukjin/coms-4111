Circuit Breaker

3. Microservices, Caching & Consistency

TL;DR: A single request fans out to many microservices in parallel. Caches (Memcached) sit in front of databases (Cassandra). The big tradeoff: strong consistency is slow, eventual consistency is fast but stale.

Parallel fan-out: one request triggers many simultaneous microservice calls

The Thundering Herd: When a failed service recovers, all backed-up requests hit it at once. Defenses: load shedding (drop low-priority requests) and exponential backoff with jitter (stagger retries randomly).

The consistency spectrum: eventual → session-level → strong

Consistency LevelGuaranteeTrade-off
EventualData converges eventually; reads may be staleFast, highly available
Session-levelYou always see your own writesGood middle ground
StrongAll reads see the latest write, alwaysSlow, requires coordination

4. CDN, Replication & Infrastructure

TL;DR: CDN boxes inside ISPs deliver video locally. Raft consensus elects leaders and ensures data survives node failures. Kubernetes orchestrates containers with auto-scaling and self-healing.

CDN — Petflix Box

  • VMAF perceptual quality metric

Raft Consensus

  • Three states: Leader, Follower, Candidate
  • Heartbeat timeout triggers election
  • Majority vote to become leader
  • Majority ACK for committed writes

VMs

Full OS isolation via hypervisor. Heavy but strong security boundaries. Each VM has its own kernel.

Containers

Share host OS kernel. Lightweight, fast to start. Weaker isolation but much more efficient.

Kubernetes (K8s) — four jobs: Schedule containers onto machines, Scale instances up/down based on load, Heal by restarting failed containers, Deploy new versions with zero downtime.
Noisy Neighbor: In shared environments, one workload can hog CPU or memory and degrade performance for others. VMs provide stronger isolation against this than containers.

6. The ACID Properties

TL;DR: Every transaction promises four things: all-or-nothing (Atomicity), valid state transitions (Consistency), no interference between concurrent transactions (Isolation), and permanence after commit (Durability).

ACID — the four guarantees that make transactions reliable

PropertyWhat it meansEnforced by
AtomicityAll or nothing — no partial transactionsWrite-Ahead Logging (WAL)
ConsistencyData stays valid (constraints hold)Application logic + locking
IsolationConcurrent transactions don't interfereSerializability / 2PL
DurabilityCommitted = permanent, survives crashesWrite-Ahead Logging (WAL)
The bank interest problem: 1,000 accounts need 6% interest applied. Crash after updating 500 accounts = 500 customers get interest, 500 don't. Atomicity prevents this: either all 1,000 get interest, or none do.

7. Write-Ahead Logging (WAL)

TL;DR: Before changing data, write a log record describing the change. The log is the source of truth. Two rules ensure correct recovery: flush all logs before acknowledging COMMIT, and flush logs before writing dirty pages.

The WAL commit protocol: log first, data second

Log record format: [TxnID, PageID, Offset, OldValue, NewValue]
UNDO: use OldValue to reverse uncommitted changes.
REDO: use NewValue to reapply committed changes.

The Two Critical Rules

Rule 1 (Durability)

Before acknowledging COMMIT, flush ALL of that transaction's log records to disk. Ensures committed changes can always be redone.

Rule 2 (Atomicity)

Before writing a dirty data page to disk, flush ALL log records for changes on that page. Ensures uncommitted changes can always be undone.

Recovery after crash

COMMIT record found

REDO all changes. The transaction committed, so its effects must be permanent (durability).

No COMMIT record

UNDO all changes. The transaction didn't finish, so roll everything back (atomicity).

Why WAL is fast: The log is append-only (sequential writes). Random writes: ~100,000 seconds for 1M pages. Sequential writes: ~1 second. The logging overhead is essentially free.

8. Concurrent Transactions & Scheduling

TL;DR: We can't run transactions one at a time (too slow). Interleaving them is risky: some interleavings produce results that match no serial order. A serializable schedule is one whose result equals some serial execution.

Serial schedules are always correct but slow. Interleaved schedules are fast but may be incorrect.

Serial Schedule

One transaction at a time, no interleaving. Always correct. Terrible performance.

Interleaved Schedule

Operations mixed between transactions. Good performance. May or may not be correct.

The three conflict types

Two operations from different transactions on the same data item where at least one is a write:

TypePatternWhy it matters
Read-Write (RW)T1 reads X, T2 writes XOrder determines what T1 sees
Write-Read (WR)T1 writes X, T2 reads XOrder determines if T2 reads old or new value
Write-Write (WW)T1 writes X, T2 writes XOrder determines final value of X
Read-Read is NOT a conflict. Two reads on the same item by different transactions never cause problems — reads don't change data, so their order doesn't matter.
Conflicts vs. Anomalies: A conflict is a potential problem (two operations that could cause issues). An anomaly is an actual problem (a conflict that led to incorrect results). Not every conflict causes an anomaly, but every anomaly arises from a conflict.

9. Conflict Serializability & Conflict Graphs

TL;DR: Build a conflict graph (nodes = transactions, edges = conflicts). No cycle = serializable (safe). Cycle = not serializable (bad). Topological sort of acyclic graph gives the equivalent serial order.

Acyclic conflict graph = safe schedule. Cyclic = unsafe, DBMS must intervene.

The 3-step test

  1. Find conflicts — scan for RW, WR, WW pairs on the same data item between different transactions
  2. Build conflict graph — draw edge from Ti → Tj if Ti's conflicting operation comes first in the schedule
  3. Check for cycles — acyclic = conflict serializable; topological sort gives an equivalent serial order

5-transaction example: conflicts identified → acyclic graph → equivalent serial order T3, T1, T4, T2, T5

Why cycles mean non-serializability: Each edge Ti → Tj means "Ti must come before Tj." A cycle Ti → Tj → Ti would require Ti before Tj AND Tj before Ti — a logical impossibility. No serial schedule can exist.
DAGs & Topological Sort: An acyclic conflict graph is a DAG (Directed Acyclic Graph). A topological ordering places nodes so all edges point forward. Algorithm: repeatedly pick a node with no incoming edges, remove it, repeat. The resulting order is a valid equivalent serial schedule. Multiple valid orderings may exist.
