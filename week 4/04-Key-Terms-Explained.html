The Basics

1. Transaction
A logical unit of work consisting of one or more database operations (reads and writes) that must either all succeed or all fail.

Think of a transaction as one complete task. For example, transferring money from account A to account B is one transaction — it involves reading A, subtracting money, writing A, reading B, adding money, writing B. All six steps are a single "unit." If any step fails, the whole thing gets rolled back as if it never happened.

Ordering a meal at a restaurant. You order an appetizer, a main, and a dessert. Either the kitchen makes the whole meal, or if they run out of a key ingredient halfway, you cancel the entire order — you don't get a half-finished meal.
2. Schedule
A sequence of operations (reads and writes) from one or more transactions, where each transaction's internal operation order is preserved.

A schedule is just the timeline of what actually happens in the database. It lists every read and write from every transaction, in the order they execute. The only rule: within a single transaction, the operations must stay in the original order. But operations from different transactions can appear in any relative order.

Example

If T1 does r1(A), w1(A) and T2 does r2(A), w2(A), then one possible schedule is:
r1(A), r2(A), w1(A), w2(A)
T1's operations are still in order (r1 before w1). T2's are still in order (r2 before w2). They are just interleaved.

Types of Schedules

3. Serial Schedule
A schedule in which transactions run one at a time, with no interleaving. All operations of one transaction complete before the next transaction begins.

A serial schedule means no multitasking. Transaction T1 runs completely, then T2 runs completely, then T3, and so on. There is zero overlap. This is always correct — since nothing interferes with anything else — but it is very slow in practice because the CPU sits idle while waiting for disk reads.

Example

r1(A), w1(A), r1(B), w1(B), r2(A), w2(A), r2(B), w2(B)
All of T1 first, then all of T2. No mixing.

A single checkout lane at a grocery store. One customer finishes completely before the next one starts. Safe, but the line gets very long.
4. Interleaved Schedule
A schedule where operations from different transactions are mixed together, while still preserving each transaction's internal order.

This is the normal case in real databases. T1 does some work, then T2 does some work, then T1 picks up where it left off, etc. This is fast (because the CPU can do useful work while one transaction waits for disk) but risky — some interleavings produce wrong results.

Example

r1(A), r2(A), w1(A), w2(A), r1(B), r2(B), w1(B), w2(B)
T1 and T2 operations are woven together.

5. Equivalent Schedules
Two schedules are equivalent if they produce the same final database state for every possible initial database state.

Two schedules are equivalent if, no matter what data the database starts with, both schedules leave the database in the exact same state at the end. It is not enough to check just one starting point — they must match on all possible starting data.

Don't confuse "same result on one test case" with true equivalence. Two schedules might give the same answer when A=50 and B=200, but different answers when A=100 and B=300. True equivalence requires matching on ALL initial states.
6. Serializable Schedule
An interleaved schedule that is equivalent to some serial schedule. The final result is the same as if the transactions had run one at a time in some order.

This is the gold standard for correctness. An interleaved schedule is serializable if its final result matches what you would get from running the transactions one at a time. It doesn't matter which serial order it matches — T1-then-T2 or T2-then-T1 — as long as it matches at least one.

Serializable

Interleaved, but result = some serial order. Safe.

Not Serializable

Interleaved, and result matches NO serial order. Dangerous.

You and a friend cook dinner together, splitting up tasks. If the meal turns out exactly like it would have if either of you had cooked alone start-to-finish, then your collaboration was "serializable" — no harm done by working in parallel.

Conflicts

7. Conflict
Two operations conflict if and only if: (1) they belong to different transactions, (2) they access the same data item, and (3) at least one of them is a write.

A conflict is a pair of operations where order matters. If you swapped the order of these two operations, the database could end up in a different state. Two reads on the same item are fine (reading doesn't change anything), but the moment a write is involved, swapping the order can change the outcome.

Two people looking at the same whiteboard is fine. But if one person is erasing and rewriting while the other is reading, the order matters a lot — the reader gets completely different information depending on whether they look before or after the rewrite.
8. Conflict Types (RW, WR, WW)
The three possible conflict patterns between operations from two different transactions on the same data item.

Read-Write (RW)

Ti reads X, then Tj writes X.

Plain: "I read it, then you changed it." If swapped, I would have read your new value instead.

Write-Read (WR)

Ti writes X, then Tj reads X.

Plain: "I changed it, then you read it." If swapped, you would have read the old value.

Write-Write (WW)

Ti writes X, then Tj writes X.

Plain: "I wrote, then you overwrote." If swapped, my value would be the final one instead of yours.

What about Read-Read?

NOT a conflict. Two reads don't change anything. Swap them all you want — the database state is identical either way. This is why the definition says "at least one must be a write."

9. Anomaly
An anomaly occurs when a conflict, given the specific ordering in a schedule, leads to an incorrect (non-serializable) result.

A conflict is a potential problem. An anomaly is when that potential problem actually causes damage. Every anomaly comes from a conflict, but not every conflict turns into an anomaly. A schedule can have many conflicts and still be perfectly safe — as long as those conflicts don't form a contradictory cycle.

Conflict

"These two operations could cause a problem if ordered badly."
Potential danger.

Anomaly

"These two operations did cause a problem in this specific schedule."
Actual damage.

A conflict is like two cars approaching an intersection at the same time — they could crash. An anomaly is when they actually crash. A traffic light (concurrency control) can ensure the conflict never becomes an anomaly.

Conflict Equivalence & Serializability

10. Conflict Equivalence
Two schedules are conflict equivalent if they involve the same set of transactions and every pair of conflicting operations appears in the same relative order in both schedules.

Two schedules are conflict equivalent if you can transform one into the other by only swapping adjacent non-conflicting operations. In other words, every conflict "points the same direction" in both schedules. The operations that don't matter (non-conflicting ones) might be in different positions, but the ones that do matter (conflicting ones) are ordered identically.

How to think about it

Imagine all the operations on a timeline. You can freely slide non-conflicting operations past each other (like sliding cards in a deck). If, after sliding, you end up with a serial schedule, then the original schedule is conflict equivalent to that serial schedule.

Conflict equivalence is stricter than general equivalence. Two schedules can produce the same result (be "equivalent") without being conflict equivalent. Conflict equivalence means the ordering of every conflicting pair is preserved, which is a structural property — not just a "same final answer" check.
11. Conflict Serializability
A schedule is conflict serializable if it is conflict equivalent to some serial schedule — that is, it can be transformed into a serial schedule by swapping only adjacent non-conflicting operations.

This is the practical test databases actually use. Instead of checking "does this schedule produce the same result as a serial one?" (which is impossible to check for all inputs), we ask: "can we rearrange this schedule into a serial one by only moving operations whose order doesn't matter?" If yes, it is conflict serializable.

The shortcut

You don't actually have to do all the swapping. The key theorem says:

A schedule is conflict serializable ↔ its conflict graph has no cycles.

Build the graph, check for cycles. That's it.

Conflict serializability is a sufficient condition for serializability, but it is slightly conservative. There exist schedules that are serializable (produce correct results) but are NOT conflict serializable. The database might reject these "technically safe" schedules. This is a deliberate trade-off: it's better to be safe and occasionally reject a valid schedule than to accidentally accept a bad one.
All Serial Schedules ⊂ Conflict Serializable Schedules ⊂ Serializable Schedules
(Every serial schedule is conflict serializable, and every conflict serializable schedule is serializable, but not vice versa.)

Wait — why are these nested circles? An intuitive walkthrough.

Serializable (biggest — "correct result" schedules)
Conflict Serializable (medium — "passes the graph test")
Serial (smallest — "no interleaving at all")

Think of it like three groups of students who passed an exam, from strictest to most lenient:

Inner circle: Serial Schedules

These are the "straight-A students who never took any risks." Zero interleaving. Always correct. Always conflict serializable too, because if there is no interleaving, there is nothing to untangle.

Example: r1(A), w1(A), r2(A), w2(A)

T1 runs fully, then T2. No mixing at all. Obviously safe. You can trivially "swap non-conflicting adjacent ops" to get a serial schedule — it already IS one.

Middle circle: Conflict Serializable (but not serial)

These are interleaved, but you can prove they are safe using the conflict graph test (no cycles). They are a superset of serial — every serial schedule is in here, PLUS some interleaved schedules that pass the graph test.

Example: T1: r1(A), w1(A)    T2: r2(B), w2(B)

Schedule: r1(A), r2(B), w1(A), w2(B)

This is interleaved (T1 and T2 ops are mixed). But T1 touches only A, T2 touches only B — they access completely different data, so there are zero conflicts, and the conflict graph has no edges at all (definitely no cycles). Conflict serializable!

Outer circle: Serializable but NOT conflict serializable

This is the tricky one. These schedules produce the correct result, but the conflict graph test says "no" (it finds a cycle). The graph test is conservative — it sometimes rejects safe schedules.

Example (the classic "blind write"):

T1: r1(A), w1(B)    T2: r2(B), w2(A)    T3: w3(A), w3(B)

Schedule: r1(A), r2(B), w3(A), w3(B), w1(B), w2(A)

The conflict graph has a cycle (T1 → T3 and T3 → T1 via different data items), so the conflict graph test says "NOT conflict serializable."

But look at what actually happens: T3 writes A and B, and then T1 and T2 overwrite both A and B anyway. T3's writes are completely obliterated — they have no lasting effect. The final state ends up identical to the serial order T3, T1, T2. The schedule IS serializable (correct result), but the conflict graph is too paranoid to see it.

The key intuition:

Serial = "I ran them one at a time." Always safe, always passes the graph test, but slow.

Conflict Serializable = "I interleaved them, but the conflict graph has no cycles, so I can prove it is safe by rearranging non-conflicting operations." Fast and provably safe.

Serializable (but not conflict serializable) = "I interleaved them, the conflict graph has a cycle, but by sheer luck the final answer is still correct." The result happens to be right, but we cannot prove it using the conflict graph method. Real databases reject these because checking true serializability for all possible inputs is too expensive. Better to be safe than sorry.

The Testing Tools

12. Conflict Graph (Precedence Graph / Serialization Graph)
A directed graph where each node represents a transaction, and a directed edge from Ti to Tj means Ti has an operation that conflicts with (and comes before) an operation in Tj.

The conflict graph is your one-stop tool for testing conflict serializability. Each edge says "this transaction must come before that one" (because of a conflict). If you can satisfy all these "must come before" constraints simultaneously, the schedule is safe. If the constraints contradict each other (a cycle), the schedule is unsafe.

How to build it

Step 1: Draw a node for each transaction (T1, T2, T3, …)
Step 2: For every pair of conflicting operations, draw an arrow from the transaction that goes first to the transaction that goes second.
Step 3: Check for cycles. No cycles = conflict serializable. Any cycle = NOT conflict serializable.

Think of it as a "who must go before whom" chart. Each arrow says "this person promised to go before that person." If person A promised to go before B, and B promised to go before A, you have a contradiction — a cycle — and no arrangement works.
13. DAG (Directed Acyclic Graph)
A directed graph that contains no cycles. If the conflict graph is a DAG, the schedule is conflict serializable.

DAG just means a graph with arrows where you can never follow the arrows and get back to where you started. When your conflict graph is a DAG, that is the "all clear" signal — no contradictions exist, and you can find a valid serial ordering.

DAG (No cycles)

T1 → T2 → T3

All arrows go "forward." You can line up the transactions in order. Safe!

Not a DAG (Has a cycle)

T1 → T2 → T1

Arrows go in a circle. Impossible to order. Unsafe!

14. Topological Ordering (Topological Sort)
A linear ordering of all nodes in a DAG such that for every directed edge u → v, node u appears before v in the ordering.

Once you know the conflict graph is a DAG, topological sort tells you the equivalent serial schedule. It lines up all the transactions so that every "must come before" constraint is satisfied. The algorithm is simple: repeatedly pick a node with no incoming arrows, add it to your list, and remove it from the graph.

Example

Graph: T1 → T2, T1 → T4, T3 → T2, T2 → T5, T4 → T5

Round 1: T1 and T3 have no incoming edges. Pick T1.
Round 2: T3 has no incoming edges. Pick T3.
Round 3: T2 and T4 have no incoming edges. Pick T4.
Round 4: T2 has no incoming edges. Pick T2.
Round 5: T5 is left. Pick T5.

Result: T1, T3, T4, T2, T5

The topological ordering is not always unique. When multiple nodes have no incoming edges at the same time, you can pick any of them. Different choices give different (but equally valid) serial orderings. All of them are correct.
15. Victim Selection
The process of choosing which transaction to abort when a cycle is detected in the conflict graph, making the schedule non-conflict-serializable.

When the conflict graph has a cycle, the database must kill one transaction to break the cycle. But which one? That choice is called victim selection. Common strategies include aborting the youngest transaction, the one that has done the least work, or the one that will be cheapest to restart. The "victim" gets rolled back and retried later.

It's like a traffic jam at a circular roundabout where no one can move. Someone has to back up (abort) so the rest can proceed. The traffic officer (DBMS) picks the car that is easiest to reroute.

How It All Fits Together

The Complete Pipeline

1. Multiple transactions run concurrently, creating an interleaved schedule.

2. We identify all conflicts (RW, WR, WW pairs on the same data item from different transactions).

3. We build the conflict graph (nodes = transactions, edges = conflict orderings).

4. We check: is the graph a DAG (no cycles)?

    ✓ Yes (DAG): The schedule is conflict serializable. Do a topological sort to find the equivalent serial schedule.

    ✗ No (cycle): The schedule is not conflict serializable. Use victim selection to abort one transaction and break the cycle.
