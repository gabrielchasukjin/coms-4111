Conflict Graphs, Topological Orderings, and Putting It All Together

From acyclic graphs to serial orderings — completing the conflict serializability framework

1. Good vs. Bad Conflict Graphs

Part 9 introduced conflict graphs, where each transaction is a node and each conflict between transactions becomes a directed edge. Part 10 opens by crystallizing the key result: how do we distinguish "good" (safe) graphs from "bad" (unsafe) ones?

Serial Schedule

In a serial schedule (say T1→T2), all edges go in the same direction — from T1 to T2. The graph is trivially acyclic: you can only move forward.

This is the simplest case. No interleaving means no possibility of cycles.

Interleaved Schedules

When we interleave operations, edges might go in both directions. A good interleaving still keeps all edges going one way (acyclic). A bad interleaving creates edges in both directions between the same pair of nodes — forming a cycle.

Acyclic → safe. Cyclic → unsafe.

Theorem (Conflict Serializability): A schedule is conflict serializable if and only if its conflict graph is acyclic. This is the central theorem of the entire concurrency unit.

But why does acyclicity guarantee serializability? The connection comes from a concept in graph theory: topological orderings.

2. DAGs and Topological Orderings

To understand why acyclic conflict graphs imply serializability, we need to understand a fundamental graph theory concept: the topological ordering.

Topological Ordering: A linear ordering of all vertices in a directed graph such that for every directed edge from vertex i to vertex j, vertex i appears before vertex j in the ordering. In other words, it "flattens" the graph into a sequence that respects every edge.

Think of it this way: if there is an edge from T1 to T2, then T1 must come before T2 in any topological ordering. The topological ordering gives you a valid sequential execution that respects all the dependencies.

Key Property of DAGs

A directed acyclic graph (DAG) always has one or more topological orderings. Conversely, a topological ordering exists if and only if the graph has no directed cycles. This is a foundational result in graph theory.

Think of project planning: tasks have dependencies (you must buy decorations before you can decorate). A project plan is a DAG — and finding a valid schedule means finding a topological ordering. If you have circular dependencies (A depends on B, B depends on C, C depends on A), no valid schedule exists.

Example: Acyclic Graph

Graph with 4 nodes: 0, 1, 2, 3

Edges: 0→1, 1→2, 1→3

Possible topological orderings:

  • 0, 1, 2, 3 — valid because every edge goes left-to-right
  • 0, 1, 3, 2 — also valid because 2 and 3 have no edge between them, so either order works

Notice that multiple topological orderings can exist. Each one represents a valid serial schedule. The database can pick whichever ordering it prefers.

Example: Cyclic Graph

Same 4 nodes, but add edge: 2→3 and 3→1

Now we have a cycle: 1→2→3→1

There is no topological ordering! Node 1 must come before 2 (because of edge 1→2), 2 must come before 3 (because of edge 2→3), but 3 must come before 1 (because of edge 3→1). This is impossible to satisfy — a contradiction.

No topological ordering means no valid serial schedule that preserves all the conflict orderings. This is precisely why a cycle in the conflict graph means the schedule is not conflict serializable.

3. Connection to Conflict Serializability

Now we can complete the logical chain that connects conflict graphs to serializability:

1 Build the conflict graph

For a given schedule, identify all conflicts (RW, WR, WW on the same data item between different transactions). Each conflict becomes a directed edge from the transaction that acts first to the one that acts second.

2 Check for cycles

If the conflict graph is acyclic (a DAG), then a topological ordering exists.

3 Topological ordering = serial order

In the conflict graph, a topological ordering of the nodes corresponds to a serial ordering of the transactions. This serial order has the same conflict structure as the original interleaved schedule — making them conflict equivalent.

The Complete Argument

Acyclic conflict graph → topological ordering exists → that ordering is a serial schedule → the original schedule is conflict equivalent to that serial schedule → the schedule is conflict serializable.

Conversely, a cycle means no topological ordering exists, so there is no serial schedule that is conflict equivalent — the schedule is not conflict serializable.

4. Example with 5 Transactions

The lecture works through a larger example to show the full three-step process in action. This is a schedule S1 with five transactions (T1 through T5) operating on five data items (A through E).

The Schedule

Schedule S1 (interleaved)

Order 12345678910
Op w1(A)r2(A)w1(B)w3(C)r2(C)r4(B)w2(D)w4(E)r5(D)w5(E)

Each operation is labeled as ri(X) or wi(X), meaning transaction i reads or writes data item X.

Step 1: Find All Conflicts

We scan through the schedule looking for pairs of operations from different transactions on the same data item where at least one is a write (RW, WR, or WW):

Conflict Data Item Operations Edge
'A' A w1(A) before r2(A) T1 → T2
'B' B w1(B) before r4(B) T1 → T4
'C' C w3(C) before r2(C) T3 → T2
'D' D w2(D) before r5(D) T2 → T5
'E' E w4(E) before w5(E) T4 → T5

Step 2: Build the Conflict Graph and Check for Cycles

Conflict Graph for S1

Nodes: T1, T2, T3, T4, T5

Edges: T1→T2 (A), T1→T4 (B), T3→T2 (C), T2→T5 (D), T4→T5 (E)

Is it acyclic? Yes! There are no cycles. Every path leads forward — you can never follow edges and return to where you started.

Verdict: The conflict graph is acyclic, therefore Schedule S1 is conflict serializable, and therefore it is serializable. The database can safely execute this interleaving.

Step 3: Find an Equivalent Serial Schedule

Since the graph is acyclic, we can find a topological ordering to produce a conflict-equivalent serial schedule. We need to respect every edge:

  • T3 must come before T2 (because of edge T3→T2)
  • T1 must come before T2 and T4 (because of edges T1→T2, T1→T4)
  • T2 must come before T5 (because of edge T2→T5)
  • T4 must come before T5 (because of edge T4→T5)

Serial Schedule S2 (conflict equivalent to S1)

T3 → T1 → T4 → T2 → T5

Expanded: w3(C), w1(A), w1(B), r4(B), w4(E), r2(A), r2(C), w2(D), r5(D), w5(E)

This serial schedule has exactly the same conflict ordering as S1 — every conflict pair appears in the same relative order. The two schedules are conflict equivalent.

This example demonstrates the power of the approach: with 5 transactions, there are 5! = 120 possible serial orderings. Instead of checking all 120, we build one conflict graph and do one cycle check to determine serializability. Then a single topological sort gives us an equivalent serial order.

5. Summary and Parting Observations

The Complete Concurrency Framework

Across Parts 5 through 10, we have built a complete framework for understanding database concurrency:

  • Concurrency is achieved by interleaving transaction operations
  • Isolation and consistency require that interleaving produces correct results
  • Serializability formalizes "correct" — an interleaved schedule is good if it is equivalent to some serial schedule
  • Conflict serializability gives us a practical test — build the conflict graph and check for cycles

Practical Observations

Flexibility

Often, many conflict-serializable schedules exist for a given set of transactions. This is good — increased flexibility and degrees of freedom mean we can choose the best-performing schedule among the valid ones.

The database scheduler has room to optimize for throughput, latency, or other objectives while still guaranteeing correctness.

Batch Size Trade-offs

How many transactions should we schedule at once? More transactions in a batch means higher concurrency and throughput, plus more possible valid schedules.

But there are costs: higher latency for individual transactions, and the scheduler itself takes longer and becomes more complex as the batch grows.

Key Takeaways from Part 10

  • ✓ A schedule is conflict serializable if and only if its conflict graph is acyclic — the central theorem
  • ✓ Topological ordering of a DAG gives a linear ordering respecting all directed edges
  • ✓ A topological ordering exists if and only if there are no directed cycles
  • ✓ In a conflict graph, topological ordering = an equivalent serial schedule
  • ✓ The 5-transaction example demonstrates the full workflow: find conflicts → build graph → check acyclicity → extract serial order
  • ✓ More transactions in a batch means more concurrency but also more complexity — a fundamental trade-off in database scheduling
