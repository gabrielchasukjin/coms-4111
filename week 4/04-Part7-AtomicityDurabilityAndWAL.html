Atomicity, Durability, and Write-Ahead Logging

How databases survive crashes without losing data or corrupting state

1. A Note: ACID is Contentious

Before diving into the mechanisms, the lecture makes an important practical observation: ACID is not universally provided. There has been — and continues to be — significant debate over whether full ACID guarantees are necessary or even desirable in all contexts.

  • Many SQL databases do not provide full ACID by default. For example, most default to "Read Committed" isolation — a weaker form that allows certain anomalies in exchange for better performance.
  • "NoSQL" databases relax ACID even more. Systems like Redis, MongoDB, Cassandra, and DynamoDB were designed for massive scale and often sacrifice strict consistency or isolation for availability and partition tolerance (the CAP theorem tradeoff).
  • "NewSQL" databases are reintroducing ACID to NoSQL-style architectures. Systems like CockroachDB and Google Spanner aim to provide both the scalability of NoSQL and the correctness guarantees of ACID.
Understanding ACID is essential not because every system provides it, but because you need to know what guarantees your system actually offers — and what anomalies are possible when those guarantees are relaxed. The mechanisms we study here (WAL, serializability) are the building blocks that make ACID possible when you need it.

2. The Goal: Ensuring Atomicity and Durability

This part of the lecture focuses on the A and D of ACID — Atomicity and Durability — and introduces the core mechanism that provides both: Write-Ahead Logging (WAL).

Atomicity

Transactions should either happen completely or not at all. If there is an abort or crash during a transaction, no effects should be seen.

Example: TXN 1 is in progress when a crash occurs. After recovery, none of TXN 1's changes are persisted — it is as if TXN 1 never started.

Durability

If the database stops running, changes due to completed transactions should all persist. Just store on stable disk.

Example: TXN 2 completed before the crash. After recovery, all of TXN 2's changes are still present.

The lecture focuses on how to accomplish atomicity via logging. Durability is conceptually simpler (write to disk), but the two are deeply intertwined because the logging mechanism that enables atomicity also enables durability.

3. Basic Idea: Physical Logging

Database tables themselves have no notion of history. Once you overwrite a value, the old value is gone. So we introduce a notion of history using a log.

Log: An ordered list of update records for ongoing transactions. Each log record contains UNDO information for every update, stored in the format: <TransactionID, location, old data, new data>. This is sufficient to undo any transaction.

Key properties of the log:

  • The database owns the log service for all applications and transactions
  • The log is logically (and usually physically) separate from the actual data
  • Logging is transparent to the application or transaction — the programmer does not manage it
  • Writes to the log are sequential, and the log can be flushed (force-written to disk)

How Logging Works: A Concrete Example

Consider a transaction T that reads A (currently 0) and writes A = 1:

T: R(A=0), W(A=1)

When T writes A = 1, two things happen in main memory:

  1. The data page for A is updated: A changes from 0 to 1 in the memory buffer
  2. An update record is written to the in-memory log: <Tid, &A, 0, 1> — meaning "transaction Tid changed A at address &A from 0 to 1"

At this point, both the updated data and the log record exist only in main memory. The disk still has A = 0 and an empty log. If the system crashes now, everything in memory is lost — but since nothing was committed, that is fine (atomicity is preserved trivially).

Why not just keep everything in memory until commit?

In theory, if we never wrote partial results to disk, atomicity would be trivial — a crash would simply lose all in-progress work. But this does not work in practice:

  • Memory constraints: what if the transaction modifies more data than fits in RAM?
  • Time constraints: what if the transaction takes hours (like the monthly interest example)?

We need to write partial results of transactions to disk. And so we need a log to be able to undo those partial results if the transaction does not complete.

4. Write-Ahead Logging (WAL) Commit Protocol

The critical question is: in what order do we write things to disk? The WAL protocol defines a precise ordering that guarantees both atomicity and durability.

Write-Ahead Logging (WAL): For each record update, write an Update Record into the LOG. Then follow two flush rules:
  • Rule 1: Flush the Update Record into the LOG before the corresponding data page goes to storage. → Enables Durability
  • Rule 2: Before a transaction commits: flush all Update Records to LOG, then flush a COMMIT Record to LOG. → Enables Atomicity

A transaction is committed once its COMMIT record is on stable storage.

Think of the WAL like a detailed receipt system. Before you move any merchandise (data) out of the warehouse (memory) to the store floor (disk), you first write a receipt (log record) saying what was moved and where it was. Before you finalize a sale (commit), you make sure all receipts are filed (flushed to disk). If anything goes wrong, you can use the receipts to undo the entire transaction.

The key insight is that the log is written to disk before the data — hence "write-ahead" logging. This means:

  • If we crash after the log is on disk but before the data is on disk, we can redo the changes using the log
  • If we crash before the log is on disk, the transaction was never committed, so no changes need to persist (atomicity)
  • The COMMIT record on disk is the definitive answer to "did this transaction complete?"

5. Why Order Matters — Incorrect Protocols

To understand why WAL works, it is instructive to see what goes wrong with other approaches. The lecture walks through two incorrect commit protocols:

Incorrect Protocol #1: Commit Before Writing Anything to Disk

Commit before writing either data or log to disk. The data (A=1) and log exist only in memory.

Problem: If we crash now, both the data and the log are lost. The transaction was "committed" but its effects have vanished.

Result: Lost T's update! Durability is violated.

Incorrect Protocol #2: Commit After Writing Data But Before Log

Write data to disk (A=1 on disk), commit, but do not write the log to disk.

Problem: The data is durable — A=1 survives a crash. But the log is lost, so after recovery, we have no record of whether T committed or was still in progress.

Result: How do we know whether T was committed?? We cannot distinguish a completed transaction from a half-finished one that needs to be rolled back.

Why WAL Gets It Right

WAL avoids both problems by committing after writing the log to disk but potentially before writing the data to disk. The log on disk serves as the authoritative record of what happened:

  • If the log contains a COMMIT record for T → T is committed, and we can redo any data changes that did not make it to disk
  • If the log does NOT contain a COMMIT record for T → T is not committed, and we can undo any data changes that DID make it to disk

6. Bank Interest Example — Full Run, Crash, and Recovery

The lecture brings back the monthly interest example from Part 5 to show WAL in action with a realistic scenario.

Full Run (No Crash)

T-Monthly-423: Apply 10% Interest to 10M Accounts

The WAL records every single update as the transaction processes accounts:

  • T-Monthly-423 START TRANSACTION
  • T-Monthly-423, 3001, 500, 550 (Account 3001: $500 → $550)
  • T-Monthly-423, 4001, 100, 110 (Account 4001: $100 → $110)
  • ... (millions more update records)
  • T-Monthly-423, 50002, 20, 22 (Account 50002: $20 → $22)
  • T-Monthly-423 COMMIT

The COMMIT record at the end confirms the transaction completed successfully. All changes are durable.

With Crash (No COMMIT Record)

Now suppose a network outage occurs at 10:29 AM — about 6 hours into the 24-hour transaction. The WAL on disk has update records for accounts processed so far, but no COMMIT record (because the transaction did not finish).

The Ambiguous State

When the system comes back at 10:45 AM, the data on disk is in a messy state. Some accounts have their interest (3001 = $550), others do not (50002 = $22). Even worse, some accounts might have been modified by other transactions during the outage window.

Without the WAL, we would have no idea which accounts were updated by T-Monthly-423 and which were not. But the WAL tells us exactly: the log contains update records for every change T-Monthly-423 made, and since there is no COMMIT record, all of those changes must be undone.

Recovery Process

1 Undo uncommitted transactions

Scan the WAL for transactions that have a START but no COMMIT. For each such transaction, use the "old data" values in the log records to restore every modified item to its original value. Account 3001 goes back to $500, Account 4001 goes back to $100, and so on. Notify developers about the aborted transaction.

2 Redo committed transactions

For any transaction that does have a COMMIT record in the log, use the "new data" values to ensure all changes made it to the data on disk. (The data might already be correct, but this step guarantees it.)

3 Back in business

After undo and redo, the database is back in a consistent state. The monthly interest transaction can be re-run from scratch.

The COMMIT record is the single source of truth. Its presence or absence on disk is the definitive answer to "did this transaction complete?" This binary check makes recovery straightforward and unambiguous, regardless of how complex the transaction was or how many data pages were modified.

7. Why Sequential Writes Matter — WAL and Performance

WAL is not just about correctness — it also provides a significant performance advantage.

Why is WAL good for performance?

  1. Updates to the WAL are sequential. Log records are appended one after another, in order. Sequential writes to disk are dramatically faster than random writes — on magnetic disks, the difference can be 100x or more.
  2. Data flushes can be done lazily. Since the log contains all the information needed to reconstruct any committed transaction, the actual data pages in the tables can be flushed to disk later, at a convenient time. This allows the system to batch and reorder data writes for efficiency.

The Speedup: 100,000 Seconds vs. 1 Second

Consider the monthly interest transaction updating 10 million bank accounts. Writing 10M individual updates to random disk locations takes approximately 10M × 10ms = 100,000 seconds (~28 hours) on a magnetic disk.

But writing 10M sequential log records to the WAL? That takes approximately 1 second. The data pages can then be flushed lazily in the background. The commit itself only requires the sequential WAL write to complete — a 100,000x speedup for the commit operation.

Sequential writes are critically important for both flash storage (SSDs) and magnetic disks, for different reasons. On magnetic disks, sequential access avoids costly seek times. On SSDs, sequential writes align with the erase-block structure and reduce write amplification. Future lectures will explore the underlying storage physics in detail.

Summary

  • ✓ ACID is contentious — many SQL databases default to weaker isolation; NoSQL databases relax ACID further; NewSQL tries to bring ACID back at scale.
  • ✓ Atomicity and Durability are achieved through logging. Atomicity means "all or nothing"; durability means "committed data survives crashes."
  • ✓ Physical logging records <TxnID, location, old data, new data> for every update, providing enough information to undo or redo any transaction.
  • ✓ We must write partial results to disk because memory is limited and transactions can be long-running. This means we need a log to undo those partial results if the transaction fails.
  • ✓ WAL has two rules: (1) flush the log record before the data page, and (2) flush all log records + COMMIT record before declaring the transaction committed.
  • ✓ The COMMIT record on disk is the single source of truth for whether a transaction completed. Present = committed (redo if needed). Absent = not committed (undo any partial changes).
  • ✓ Incorrect protocols fail — committing before writing anything to disk loses data; committing after data but before log makes it impossible to determine transaction status on recovery.
  • ✓ Recovery uses undo and redo: undo uncommitted transactions (restore old values), redo committed transactions (apply new values), then resume normal operation.
  • ✓ WAL is great for performance — sequential log writes are ~100,000x faster than random data writes. Data pages can be flushed lazily after commit.
