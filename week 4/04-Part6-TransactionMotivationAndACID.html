Transaction Motivation and ACID Properties

Why transactions exist and the four guarantees every database should provide

1. Motivation for Transactions

Grouping user actions (reads and writes) into transactions helps with two fundamental goals that every reliable database must achieve:

Goal 1: Recovery and Durability

Keeping the database data consistent and durable in the face of crashes, aborts, and system shutdowns. The database may experience power outages at any moment. Individual transactions may be aborted by the user. We need a way to ensure that transactions are either durably stored in full, or not at all — and we need a log to be able to "roll back" incomplete transactions.

Goal 2: Concurrency

Achieving better performance by parallelizing transactions without creating anomalies. Disk accesses are frequent and slow, so we optimize for throughput (number of transactions per second) by interleaving work. But users should still be able to execute transactions as if in isolation, so that consistency is maintained.

These two goals correspond to different ACID properties. Recovery and durability map to the A (Atomicity) and D (Durability). Concurrency maps to I (Isolation). Together with C (Consistency), they form the complete set of guarantees that transactions provide.

2. Protection Against Crashes and Aborts

To see why transactions matter, consider a concrete scenario. A client executes two SQL statements that are meant to work together:

The Problem: No Transaction Boundary

INSERT INTO SmallProduct(name, price) SELECT pname, price FROM Product WHERE price <= 0.99

DELETE Product WHERE price <= 0.99

The intent is clear: move all cheap products (price ≤ $0.99) from the Product table into a SmallProduct table, then remove them from Product. But what happens if the system crashes after the INSERT but before the DELETE?

The Result: Data Duplication

The cheap products now exist in both tables. The INSERT completed and was written to disk, but the DELETE never executed. We have duplicated data with no easy way to tell which products were already moved. This is exactly the kind of partial-execution bug that transactions prevent.

The Fix: Wrap It in a Transaction

With Transaction Boundaries

START TRANSACTION

    INSERT INTO SmallProduct(name, price) SELECT pname, price FROM Product WHERE price <= 0.99

    DELETE Product WHERE price <= 0.99

COMMIT OR ROLLBACK

Now if the system crashes after the INSERT but before the DELETE, the entire transaction is rolled back on recovery. The INSERT is undone, and the database returns to its state before the transaction began. We can then safely re-run the entire transaction from scratch.

The transaction boundary tells the database: "these operations are a single unit. Either execute all of them, or none of them." This is the fundamental promise that makes it safe to compose multiple SQL statements into complex operations.

3. The Need for Concurrent Execution

Even with crash protection solved, there is a second major challenge: performance. Running transactions one at a time is far too slow for any real system.

Throughput vs. Latency: Throughput is the number of transactions completed per second. Latency is the time to complete any single transaction. When disk accesses are frequent and slow, we can improve throughput by interleaving transactions — while one transaction waits for a disk read, another can use the CPU. This trades slightly higher latency for dramatically better throughput.

The Concurrency Problem: Two Managers, One Product

Consider two managers trying to discount the same product at the same time:

Client 1

UPDATE Product SET Price = Price - 1.99 WHERE pname = 'Gizmo'

Client 2

UPDATE Product SET Price = Price * 0.5 WHERE pname = 'Gizmo'

What could go wrong? Without proper isolation, both clients might read the same original price, compute their discount independently, and then write back. One discount would be lost entirely — a classic lost update anomaly. If Gizmo starts at $10, the correct result (in either serial order) would be either $3.01 or $4.005. But with a lost update, you might get $8.01 or $5.00 — only one discount applied.

The Fix: Transaction Isolation

Each Statement Gets Its Own Transaction

Even single-statement operations are wrapped in transactions by the database. The DBMS ensures that the two UPDATEs execute as if one happened entirely before the other — even though they may be processed concurrently under the hood.

With explicit transactions, this becomes even clearer: each client wraps their UPDATE in START TRANSACTION ... COMMIT. The database's concurrency control mechanisms (locks, timestamps, or MVCC) ensure the two transactions do not interfere with each other.

Think of it like two people trying to edit the same cell in a spreadsheet. If they both read the value, change it locally, and write it back, one person's change will be lost. The solution is a lock: one person edits while the other waits, then the second person edits starting from the updated value. That is exactly what database isolation provides, but at massive scale.

4. The ACID Properties

The guarantees that transactions provide are formalized as the four ACID properties. These are the gold standard for transactional correctness, first articulated in the early 1980s and still debated today.

An important caveat

ACID continues to be a source of great debate. In fact, by default, most databases do not provide full ACID guarantees. Many systems relax isolation for performance (e.g., using "Read Committed" instead of full serializability). Understanding what each property actually means — and what you lose when it is relaxed — is essential for building correct systems.

A — Atomicity

Atomicity: A transaction's activities are all or nothing. The database state shows either all the effects of the transaction, or none of them. There is no in-between state visible to any observer.

Intuitively, in the real world, a transaction is something that would either occur completely or not at all. A bank transfer either moves the money or it does not — you never see a state where the money has left one account but not arrived in the other.

There are exactly two possible outcomes for a transaction:

  • Commit: all the changes are made permanent. The transaction succeeded.
  • Abort: no changes are made. The transaction is rolled back as if it never happened.
Think of atomicity like an email draft. You can write a long email with multiple paragraphs, but until you hit "Send," nothing is visible to the recipient. If you close the window and discard the draft, it is as if you never wrote it. Hitting "Send" is the commit — everything becomes visible at once.

C — Consistency

Consistency: The database must always satisfy user-specified integrity constraints. A transaction moves the database from one consistent state to another consistent state.

Examples of integrity constraints:

  • Primary key uniqueness — account numbers cannot be duplicated
  • Domain constraints — stock amounts cannot be negative
  • Balance invariants — the sum of all debits must equal the sum of all credits

Who is responsible for consistency?

Consistency is a shared responsibility. The programmer writes transactions that go from one consistent state to another (e.g., a transfer that debits one account and credits another by the same amount). The system ensures atomicity — so if a transaction is interrupted, partial changes do not violate constraints. Assuming the system maintains atomicity, consistency is often the user's responsibility to get right.

Real-World Consequence: The Bank of Ireland Glitch (2023)

In August 2023, a glitch at Bank of Ireland allowed customers to withdraw money beyond their account balances at ATMs. Long lines formed as people rushed to exploit the bug. This is a real-world example of a consistency violation — the system failed to enforce the constraint that withdrawals cannot exceed the available balance. Proper transaction design and constraint enforcement would have prevented this.

I — Isolation

Isolation: A transaction executes concurrently with other transactions, but the effect is as if each transaction executes in isolation of the others. Transaction A should not be able to observe intermediate changes from a concurrent transaction B during its run.

Isolation is what we explored in depth in Part 4 (Concurrency and Serializability). The core idea is that even though transactions run concurrently for performance, the result must be indistinguishable from some serial (one-at-a-time) execution. This is precisely the serializability guarantee, enforced through conflict graphs, locking protocols, or multi-version concurrency control.

Isolation is arguably the most complex and performance-sensitive of the four ACID properties. Full isolation (serializability) is expensive, which is why many databases default to weaker isolation levels that allow certain anomalies in exchange for better throughput.

D — Durability

Durability: Once a transaction has committed, its effects must persist — even after the program terminates, even after power failures, even after crashes. The data survives.

In practice, durability means:

  • Write data to disk — volatile memory (RAM) is lost on power failure, so committed data must reach non-volatile storage
  • In data center settings: replicate data across machines, maintain backups, use write-ahead logs to ensure nothing is lost

Durability is what WAL (Write-Ahead Logging) provides, as we will see in Part 7. The log is written to disk before the transaction commits, ensuring that even if the system crashes immediately after the commit, the changes can be recovered.

5. Challenges for ACID

Providing all four ACID properties simultaneously is hard. The database must handle all of the following, and it must do so with scalability and performance:

1 Power failures and crashes

The system can lose the contents of memory at any moment. Everything in RAM — including partially executed transactions — vanishes. The database must be able to recover to a consistent state using only what is on disk.

2 User aborts

Users may cancel transactions at any point. The database must be able to roll back all changes made by an aborted transaction. This requires logging what happened so the undo is possible.

3 Many users executing concurrently

Thousands of transactions may be in flight simultaneously. The database must interleave their operations for performance while ensuring that the result is equivalent to some serial execution. This is the isolation challenge.

The real engineering challenge is not just providing ACID guarantees — it is providing them at scale, with high throughput and low latency. This is why database internals are among the most sophisticated pieces of systems software ever built. The next parts of this lecture will show the specific mechanisms (WAL, locking, conflict serializability) that make it possible.

Summary

  • ✓ Transactions serve two goals: (1) recovery and durability in the face of crashes, and (2) correct concurrent execution for performance.
  • ✓ Without transactions, partial execution is dangerous — a crash between two related SQL statements can leave the database in an inconsistent state with duplicated or missing data.
  • ✓ Concurrent execution is essential for throughput — disk I/O is slow, and interleaving transactions keeps the system productive. But concurrency without isolation leads to anomalies like lost updates.
  • ✓ Atomicity — all or nothing. A transaction either commits (all changes visible) or aborts (no changes visible).
  • ✓ Consistency — transactions move the database from one valid state to another, preserving integrity constraints. This is a shared responsibility between the programmer and the system.
  • ✓ Isolation — concurrent transactions produce results as if they ran serially. This is the most complex property and the one most commonly relaxed for performance.
  • ✓ Durability — committed data survives crashes, power failures, and system restarts. Achieved by writing to disk and using replication.
  • ✓ The real challenge is providing all four ACID properties simultaneously, at scale, with high performance. This motivates the mechanisms (WAL, locking, serializability) covered in subsequent parts.
