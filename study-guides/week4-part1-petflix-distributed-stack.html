Layer 1: DNS Routing

Entry Point

From Name to Address

When you type petflix.com, your browser doesn't know where that is. DNS (Domain Name System) translates the human-readable name into an IP address like 34.120.55.9.

But Petflix's DNS does much more than a simple lookup. It makes intelligent routing decisions.

Three DNS Superpowers

🌎
Geo-Routing

Routes users to the nearest data center based on geographic location. A user in Tokyo gets routed to AP-South, not US-East.

💚
Health-Check Failover

Continuous health checks on every region. If US-East goes down, DNS automatically reroutes to the next closest healthy region.

⚖️
Weighted Load Balancing

Distributes traffic proportionally based on capacity. Not every data center is the same size.

✈️ Analogy: Airport Air Traffic Control

Think of DNS like air traffic control. Planes (requests) need to land at airports (data centers). ATC routes them to the nearest airport with capacity, reroutes around storms (outages), and balances traffic across runways (weighted load balancing).

Layer 2: API Gateway & Circuit Breaker

Gatekeeper

PetGate: The Single Entry Point

Every request to Petflix goes through the API Gateway (called "PetGate"). It is the front door — no request reaches a backend service without passing through it first.

🏥 Analogy: Hospital Front Desk

PetGate is like a hospital front desk. It checks your ID (authentication), makes sure you have an appointment (rate limiting), figures out which department you need (routing), picks the least busy doctor (load balancing), and knows which doctors are on call today (service discovery).

5 Gateway Responsibilities

🔐
Authentication — Verify the user is who they say they are
🚫
Rate Limiting — Prevent abuse; cap requests per user per second
🔌
Request Routing — Send /videos to VideoService, /users to UserService
⚖️
Load Balancing — Pick the least-loaded instance of each service
🔍
Service Discovery — Know which service instances are alive and where

The Circuit Breaker Pattern

What happens when a backend service is failing? Without protection, the gateway keeps sending requests to a dead service, wasting resources and making timeouts cascade. The Circuit Breaker pattern stops this.

💡 Key Principle

It is better to return a degraded response quickly than to wait for a perfect response that never comes.

⚡ Circuit Breaker State Machine

Step through the three states of a circuit breaker. Watch how it protects the system.

✅

CLOSED (Normal)

Requests flow through normally. The breaker monitors failure rates. If failures exceed threshold, it trips open.

⛔

OPEN (Failing)

All requests immediately return a fallback response. No traffic sent to the failing service. Timer counts down to half-open.

⚠️

HALF-OPEN (Testing)

Lets a small number of test requests through. If they succeed, close the breaker. If they fail, open it again.

State: CLOSED — all requests flowing normally

The Thundering Herd Problem

Imagine a service crashes and recovers. All the backed-up requests that were waiting slam the service simultaneously — causing it to crash again. This vicious cycle is the Thundering Herd.

🚫
Load Shedding

Intentionally drop low-priority requests when overloaded. Like electrical grid rolling blackouts — sacrifice some to save the whole.

⏲
Exponential Backoff + Jitter

Retry after 1s, 2s, 4s, 8s... with a random component so clients don't all retry at the same instant.

⚠️ Why jitter matters

Without jitter, exponential backoff alone means all clients retry at 1s, then all at 2s, then all at 4s — still a thundering herd, just slower! The random component (jitter) spreads retries across time.

Layer 4: Data & Caching

Storage

Two-Tier Architecture

Petflix uses a two-tier data architecture to balance speed and durability:

PetCache (Memcached)

~μs

In-memory key-value store. Microsecond reads. Volatile — data lost on restart. Target: 95%+ cache hit ratio.

+

Cassandra (NoSQL)

~ms

Distributed NoSQL database. Millisecond reads. Durable — data persists to disk. Handles massive write throughput.

Consistent Hashing

With multiple cache nodes, how do you know which node stores a given key? You could use hash(key) % N — but what happens when you add or remove a node?

Modular Hashing

node = hash(key) % N

Add 1 node? Nearly ALL keys get reassigned. Massive cache miss storm.

VS

Consistent Hashing

ring-based assignment

Add 1 node? Only ~1/N keys get reassigned. Minimal disruption.

🔑 Why this matters

At scale, adding a single cache node with modular hashing would cause millions of cache misses simultaneously — effectively a self-inflicted DDoS. Consistent hashing makes scaling smooth.

The Consistency Spectrum

Distributed systems face a fundamental tradeoff: how "fresh" is the data you read? Drag the slider to explore the spectrum.

Eventual Session Strong

Eventual Consistency

Replicas eventually converge to the same value, but brief stale reads are OK. Great for view counts, like counts, recommendations. This is what most large systems choose for performance.

Example: "This video has 1,234,567 views" might be off by a few hundred — nobody notices.
