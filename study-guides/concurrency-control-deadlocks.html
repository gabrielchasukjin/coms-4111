Setting the Stage

Context

Why Does This Matter?

Databases handle thousands of transactions per second. Running them one at a time (serially) is safe but painfully slow. Running them all at once is fast but can corrupt your data.

Concurrency control is the art of running transactions simultaneously while making it look like they ran one at a time.

🧱 What this builds on

You already know: transactions are groups of operations (reads/writes) that must be Atomic, Consistent, Isolated, and Durable (ACID). The Isolation part is what this entire guide is about — how do we keep transactions from interfering with each other?

The 4 Concurrency Anomalies

Problems

When transactions run concurrently without any protection, four bad things can happen. Each arises from a specific type of conflict between operations.

⚠️ Conflict vs Anomaly

A conflict is a potential problem (two ops on same data, at least one is a write). An anomaly is when that conflict actually causes incorrect behavior.

💥 W → W

Lost Update

Two transactions write the same item. The first write is silently overwritten and lost forever.

T1: read(A)=100
T2: read(A)=100
T1: write(A, 150) ← +50
T2: write(A, 80)  ← −20
Result: A=80 (T1's +50 is lost!)
🗑️ W → R

Dirty Read

You read data written by a transaction that hasn't committed yet. If it aborts, you've read garbage.

T1: write(A, 200)
T2: read(A)=200 ← reads T1's write
T1: ABORT! ← rolls back
T2 used a value that never existed!
🔀 R → W

Unrepeatable Read

You read a row, someone changes it, you read it again — different value. The same row changed.

T1: read(A)=100
T2: write(A, 200); COMMIT
T1: read(A)=200 ← different!
Same query, different answer
👻 R → W

Phantom Read

You run a query, someone inserts/deletes rows, you run it again — different set of rows.

T1: SELECT * WHERE age>20 → 5 rows
T2: INSERT (age=25); COMMIT
T1: SELECT * WHERE age>20 → 6 rows
A "phantom" row appeared!

Conflict Serializability

Detection

The Goal: Make It Look Serial

A schedule is conflict serializable if it produces the same result as some serial (one-at-a-time) execution. We check this by building a conflict graph.

Step-by-Step: Building a Conflict Graph

1
Identify conflicts between operations of different transactions on the same data item:
W-W W-R R-W R-R ← NOT a conflict!
2
Draw the graph: each transaction is a node. For each conflict, draw an edge from the earlier transaction to the later one.
3
Check for cycles:
No cycle → Conflict Serializable ✓
Cycle → NOT Serializable ✗

Two-Phase Locking (2PL)

Prevention

From Detection to Prevention

Conflict graphs tell us after the fact if a schedule was bad. But we need to prevent bad schedules from happening. Enter: locking.

✅ The core idea

Before you touch data, you must ask for permission (a lock). If someone else is using it in a conflicting way, you wait.

Two Types of Locks

🔓
Shared Lock (S)

Required before reading. Multiple transactions can hold S locks on the same item simultaneously.

🔒
Exclusive Lock (X)

Required before writing. No one else can hold any lock on the item. Totally exclusive.

Lock Compatibility Matrix

When you request a lock, can you get it?

S Lock Held X Lock Held
Request S ✓ YES ✗ WAIT
Request X ✗ WAIT ✗ WAIT
💡 Think of it like a library book

S lock = "I'm reading this book" — others can read too, but no one can scribble in it.
X lock = "I'm editing this book" — nobody else can touch it until I'm done.

Strict 2PL Rules

Rule 1: Acquire S-lock before every read
Rule 2: Acquire X-lock before every write
Rule 3: Release ALL locks only at commit or abort

Deadlocks

The Catch

Locking Introduces a New Problem

2PL prevents concurrency anomalies — great! But what happens when two transactions are each waiting for a lock the other one holds? Neither can proceed. This is a deadlock.

Classic Deadlock Scenario

🎬 The Dining Philosophers of Databases
T1 acquires X-lock on A ✓
T2 acquires X-lock on B ✓
T1 requests S-lock on B → BLOCKED (T2 holds X on B)
T2 requests S-lock on A → BLOCKED (T1 holds X on A)
💀 DEADLOCK — T1 waits for T2, T2 waits for T1. Forever.

Waits-For Graph

To detect deadlocks, we build a waits-for graph:

  • Nodes = transactions
  • Edge T1 → T2 means "T1 is waiting for a lock that T2 holds"
  • Cycle = DEADLOCK
🚨 Don't confuse the two graphs!

Conflict graph: based on the order of already-executed operations → checks serializability
Waits-for graph: based on who's waiting for whose lock right now → detects deadlocks
They are completely different graphs!

CYCLE DETECTED → DEADLOCK!

Deadlock Solutions

Strategies

Three Approaches

🔍
Detection

Let deadlocks happen, detect them, abort a victim to break the cycle.

🛡️
C2PL

Conservative 2PL: grab ALL locks at the start. Can't deadlock if you never wait incrementally.

⏰
Timestamp-Based

Wait-Die and Wound-Wait: use transaction age to decide who waits and who aborts.

Wait-Die vs Wound-Wait

Both use transaction timestamps (earlier timestamp = older = higher priority).

Wait-Die

"Old people are patient, young people die"

If T1 is OLDER → T1 waits for T2
If T1 is YOUNGER → T1 dies (aborts)

Only the requesting txn ever aborts. The one holding the lock is never forced to give it up.

VS

Wound-Wait

"Old people are aggressive, young people are patient"

If T1 is OLDER → T1 wounds T2 (forces T2 to abort)
If T1 is YOUNGER → T1 waits for T2

The older txn always wins. It can preempt the younger one's lock.

🔑 Critical detail: Timestamp preservation

When a transaction is aborted and restarts, it keeps its original timestamp. This means it gets "older" over time and will eventually have the highest priority, preventing starvation.
