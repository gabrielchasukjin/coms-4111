Layer 5: CDN (Content Delivery Network)

Delivery

Netflix Open Connect / "Petflix Box"

Netflix doesn't stream video from a single datacenter. It installs physical servers (called Open Connect Appliances) directly inside ISP data centers around the world. For Petflix, think of them as "Petflix Boxes".

📡 Why physical hardware inside ISPs?

Video files are enormous. A single 4K movie can be 15+ GB. Sending that across the internet backbone for every viewer would be slow, expensive, and congest the entire network. By putting servers inside your ISP, the video only has to travel from a server down the hall -- not across the continent.

CDN: The Core Idea

A Content Delivery Network is a system of geographically distributed servers that cache and deliver content from locations close to users. Instead of one origin server handling everything, hundreds of edge servers around the globe serve content locally.

Per-Title Encoding

Not all video content is equally complex. A slow, dialogue-driven drama has much less visual complexity than a fast-paced action scene with explosions. Netflix creates a custom encoding recipe for each title.

🎬
Slow Drama

Static shots, talking heads. Needs less bitrate to look great. More quality levels can be packed into a smaller encoding ladder.

💥
Fast Action

Explosions, rapid motion. Needs more bitrate to avoid blurriness. Encoding ladder uses higher bitrates at each step.

📈 Encoding Ladder

Each title gets its own "ladder" of quality levels (e.g., 240p at 150kbps, 480p at 750kbps, 1080p at 4800kbps). The ladder is custom per title based on its visual complexity, ensuring each step looks perceptually good.

Adaptive Bitrate Streaming (ABR)

Your bandwidth fluctuates constantly. ABR handles this gracefully:

1. Video is split into small chunks (2-10 seconds each)
2. Each chunk is encoded at multiple quality levels
3. Player monitors bandwidth in real time
4. For each chunk, player picks the best quality that fits current bandwidth
5. Quality switches happen seamlessly between chunks
⚠️ Why you sometimes see quality drops

When your WiFi stutters, the player drops to a lower quality chunk to avoid buffering. When bandwidth recovers, it ramps back up. This is ABR working as designed.

VMAF (Video Multi-Method Assessment Fusion)

How do you objectively measure video quality in a way that matches human perception?

VMAF is a perceptual quality metric (score 0-100) developed by Netflix. It uses machine learning trained on thousands of human ratings to predict how good a video looks to viewers.

0-20: Unwatchable
40-60: Acceptable
80-100: Excellent

VMAF is used to build each title's encoding ladder, ensuring that every quality step on the ladder looks perceptually good to humans, not just technically correct.

Layer 6: Pods, Leader Election & Replication

Consensus

Pods: Replicated Service Groups

A Pod is a group of replicated service instances that work together for fault tolerance. If one instance crashes, the others keep serving requests.

Leader

Handles all writes. The single source of truth. Propagates changes to followers.

Followers

Receive copies of writes. Can serve reads. Stand by to replace the leader if it fails.

🚨 The critical question

What happens when the leader fails? How do the followers agree on who becomes the new leader -- without a human intervening? This is the consensus problem.

Raft Consensus Algorithm

Raft was designed to be understandable (unlike Paxos, which is notoriously difficult). It solves leader election and log replication.

Three Node States:
Leader
Follower
Candidate
Heartbeats: Leader sends periodic heartbeats to all followers
Election timeout: Each follower has a randomized timer, resets on heartbeat
Timeout expires: Follower suspects leader is dead, becomes Candidate
RequestVote: Candidate asks all other nodes for their vote
Majority wins: If candidate gets majority, it becomes the new Leader
💡 Why majority is the magic number

Any two majorities of the same set must overlap by at least one member. This means at least one node that voted for the new leader also has the latest committed data. That guarantees no data is lost. Writes also require majority ACK before being considered committed.

🤔 Why odd-numbered clusters?

A 5-node cluster needs 3 votes (majority) and tolerates 2 failures. A 3-node cluster needs 2 votes and tolerates 1 failure. Even-numbered clusters waste a node: a 4-node cluster still needs 3 votes and only tolerates 1 failure -- same as 3 nodes!

VMs vs Containers -- Click to Explore

Click each attribute to reveal the comparison details.

💻 Virtual Machines

Full guest operating system
Strong isolation (hardware-level)
Heavy: GBs of disk space
Slow startup: seconds to minutes
Best for: security boundaries
VS

📦 Containers

Shared host kernel
Weaker isolation (process-level)
Minimal: MBs of disk space
Fast startup: milliseconds to seconds
Best for: microservices
🎯 Isolation ▼

VM: Each VM runs its own kernel and OS. A compromised VM cannot affect the host. Hardware-level isolation via hypervisor.

Container: Shares the host kernel. Uses Linux namespaces and cgroups for isolation. A kernel exploit could escape the container.

💾 Resource Usage ▼

VM: Each VM includes a full OS (1-10+ GB). Running 10 VMs = 10 copies of the OS. Significant memory and disk overhead.

Container: Only packages the app and its dependencies (10-100 MB). The OS is shared. You can run hundreds of containers where 10 VMs would fit.

⏱ Startup Time ▼

VM: Must boot an entire OS -- BIOS, kernel, init system, services. Takes 30 seconds to several minutes.

Container: Just starts a process (the kernel is already running). Takes milliseconds to a few seconds. Critical for auto-scaling.

💪 Best Use Case ▼

VM: Multi-tenant cloud (AWS), running untrusted workloads, legacy apps that need a full OS, strict security boundaries.

Container: Microservices, CI/CD pipelines, rapid scaling, dev/staging environments. Anywhere speed and density matter.

The Noisy Neighbor Problem

When multiple workloads share the same physical machine, one workload can hog resources (CPU, memory, disk I/O, network) and degrade performance for everything else running alongside it.

🎉 Real-world analogy

Apartment building: your neighbor throws a party that shakes the walls, clogs the elevator, and fills the parking lot. You can't sleep, even though you're paying the same rent. This is the noisy neighbor problem in cloud computing.

Kubernetes (K8s)

Kubernetes is an open-source container orchestration platform, originally based on Google's internal system called Borg. It automates the deployment, scaling, and management of containerized applications.

Think of K8s as the operating system for your cluster -- it decides what runs where, and keeps everything healthy.

Layer 8: Cores, Threads & Video Decoding

Bare Metal

Concurrency vs Parallelism

These sound similar but are fundamentally different:

Concurrency

Dealing with multiple tasks at once. One core rapidly switches between tasks (interleaving). The tasks make progress, but only one actually runs at any instant.

1 chef juggling 4 dishes -- flipping between them, never truly doing two at once.
VS

Parallelism

Actually doing multiple tasks simultaneously. Multiple cores each execute a different task at the same physical instant.

4 chefs, each working on their own dish at the same time.

Race Conditions

When two threads access shared data and the result depends on timing, you have a race condition. The classic example: both threads try to increment a shared counter.

Locks (Mutex)

The fix for race conditions: mutual exclusion. Only one thread can hold the lock at a time. Any other thread that wants access must wait.

// With a mutex lock:
Thread A: lock() // acquired
Thread A: read(counter) = 10
Thread B: lock() // BLOCKED -- waits
Thread A: write(counter = 11)
Thread A: unlock() // released
Thread B: lock() // now acquired
Thread B: read(counter) = 11
Thread B: write(counter = 12)
Thread B: unlock()
// Result: counter = 12 (correct!)

Deadlocks

Two threads each hold a resource that the other needs. Neither can proceed. Classic circular wait.

💀 Thread Deadlock
Thread A: lock(Resource 1) -- acquired
Thread B: lock(Resource 2) -- acquired
Thread A: lock(Resource 2) -- BLOCKED (B holds it)
Thread B: lock(Resource 1) -- BLOCKED (A holds it)
Neither can make progress. Deadlock.

Hardware Video Decoders

Your device has fixed-function silicon (dedicated hardware circuits) specifically designed to decode video. This is vastly more power-efficient than using the general-purpose CPU.

Software Decode (CPU)

General-purpose CPU runs decode instructions. Flexible but power-hungry and slow for video. Drains battery fast.

Hardware Decode (ASIC)

Dedicated silicon circuits built specifically for video decode. Extremely efficient -- uses a fraction of the power. This is why your phone can play 4K video for hours.
