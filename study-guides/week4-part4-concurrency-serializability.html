Setting the Stage

Context

Where We Left Off

WAL (Write-Ahead Logging) gave us Atomicity and Durability -- two pillars of ACID. But real databases serve thousands of users simultaneously. We can't run transactions one at a time -- it would be painfully slow.

This lecture is all about the I in ACID: Isolation. How do we interleave operations from multiple transactions for performance while still producing correct results?

🍳 Restaurant Kitchen Analogy

Imagine a restaurant kitchen preparing multiple orders at once. The chef interleaves steps -- while one steak is grilling, they're plating a salad. But they must never mix ingredients between orders. Your pasta sauce should never end up on someone else's burger. That's isolation in a nutshell.

Schedules

Definitions

What is a Schedule?

A schedule is a sequence of operations (reads and writes) from multiple transactions, where the operations of each individual transaction appear in their original order.

Types of Schedules

Serial Schedule

No interleaving at all. One transaction completes entirely before the next starts. Always correct, but slow.

T1: r(A) w(A) r(B) w(B)
T2:                   r(A) w(A) r(B) w(B)
Interleaved Schedule

Operations from different transactions are mixed together. May or may not be correct. Fast.

T1: r(A) w(A) ··· r(B) w(B)
T2:           r(A) w(A) ··· r(B) w(B)

Key Definitions

≡
Equivalent schedules: Two schedules are equivalent if they produce the same final state for every possible initial database state.
✔
Serializable schedule: An interleaved schedule that is equivalent to some serial schedule. Any serial order is fine -- we just need it to match at least one.
🏆 Gold Standard

Serializability is the gold standard for correctness. If a schedule is serializable, we can guarantee it's correct -- even though operations were interleaved for performance.

Conflicts & Anomalies

Danger Zone

What is a Conflict?

Two operations conflict if all three conditions hold:

1
They belong to different transactions
2
They access the same data item
3
At least one of them is a write
W-W conflict W-R conflict R-W conflict R-R = NOT a conflict!
📝 Google Docs Analogy

If two people are reading a Google Doc at the same time, no problem. But if one person is editing while another reads or edits the same paragraph? Potential chaos.

Conflict vs. Anomaly -- The Critical Distinction

🚨 Don't Confuse These!

A conflict is a potential problem -- two operations that could interfere.
An anomaly is an actual problem -- the conflict produced an incorrect result.

Not every conflict causes an anomaly, but every anomaly arises from a conflict. Think of it like a wet floor: it's a hazard (conflict), but you only have a problem if someone actually slips (anomaly).

⚠️ Why This Matters

Conflict serializability is conservative: it may reject some schedules that are actually correct (because they have conflicts that don't cause anomalies). But it will NEVER accept an incorrect schedule. Safety first!

Conflict Serializability

The Test

How to Test: Build a Conflict Graph

1
Create a node for each transaction in the schedule.
2
For each conflict, draw a directed edge from the transaction whose operation came first to the one whose operation came second.
3
Check for cycles:
No cycle → Conflict Serializable ✔
Cycle → NOT Serializable ✘

Why Cycles Mean Non-Serializability

If T1 → T2 → T1 exists in the conflict graph, it means:

  • T1 must come before T2 (because of the first edge)
  • T2 must come before T1 (because of the second edge)
  • T1 must be both before AND after T2 -- a logical contradiction!
🍳 Breakfast/Lunch Analogy

It's like saying "I need to eat breakfast before lunch" AND "I need to eat lunch before breakfast." Both can't be true. There is no valid linear ordering.

⚡ Performance Note

Cycle detection in a directed graph runs in O(V + E) time using DFS -- very efficient! V = number of transactions, E = number of conflict edges.

DAGs & Topological Orderings

Theory

DAG = Directed Acyclic Graph

When a conflict graph has no cycles, it's a DAG. This is the "safe" case -- the schedule is conflict serializable.

From a DAG, we can find a topological ordering: a linear ordering of all nodes where every edge points "forward" (from earlier to later in the ordering).

Topological Sort Algorithm

1
Find a node with no incoming edges (in-degree = 0)
2
Add that node to the ordering
3
Remove the node and all its outgoing edges from the graph
4
Repeat until all nodes are placed
⚠️ Important: Not Unique!

If multiple nodes have in-degree 0 at the same step, you can pick any of them. This means the topological ordering may not be unique -- there can be multiple valid equivalent serial schedules.
