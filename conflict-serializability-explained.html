Conflict Serializability, Step by Step

What it tests, how to find conflicts, and how to detect cycles

1. What Is This Question Actually Testing?

The core question: If two transactions ran at the same time (interleaved), is the result guaranteed to be the same as if they had run one after the other (serially)? If yes, the schedule is safe. If no, the database could produce corrupted or inconsistent data.

Imagine two people editing the same Google Doc at the same time. If the final document looks like Person A edited first, then Person B edited — or Person B first, then Person A — it's fine. But if the final document is some weird mix that matches neither ordering, something went wrong. That's what serializability checks for.

Conflict Serializable: A schedule (interleaved execution) is conflict serializable if you can rearrange it into a serial order (T1 then T2, or T2 then T1) by swapping only non-conflicting operations. If yes, it's safe. If not, it could produce incorrect results.

2. What Counts as a Conflict?

Two operations conflict if ALL three conditions are true:

  1. Different transactions (T1 and T2)
  2. Same data item (both touch A, or both touch B)
  3. At least one is a Write
Two Reads are NEVER a conflict. Reading doesn't change anything, so the order doesn't matter. The moment a Write is involved, order matters because someone is changing the data.

There are only 3 types of conflicts:

Type Pattern Why it matters
Read-Write (RW) T1:R(A) ... T2:W(A) T1 read a value that T2 is about to change
Write-Read (WR) T1:W(A) ... T2:R(A) T2 reads a value that T1 just changed
Write-Write (WW) T1:W(A) ... T2:W(A) Both overwrite the same item — final value depends on order

Not a conflict: T1:R(A) ... T2:R(A) — both are just reading. Order doesn't matter.

Not a conflict: T1:W(A) ... T2:W(B) — different data items. They don't interfere.

3. Walkthrough: The Exam Question

The Schedule

Here's the interleaved execution. Read it left to right — that's the time order:

Time → 12345678
T1 R(A)W(A)R(B)W(B)
T2 R(A)W(A)R(B)W(B)

Step 1: Find All Conflicts

Go through every pair of operations from different transactions on the same item where at least one is a Write. The one that happens first in time goes on the left.

On item A:

T1:R(A) → T2:W(A)   RW T1 reads A at time 1, T2 writes A at time 4
T2:R(A) → T1:W(A)   RW T2 reads A at time 2, T1 writes A at time 3
T1:W(A) → T2:W(A)   WW T1 writes A at time 3, T2 writes A at time 4

On item B:

T2:R(B) → T1:W(B)   RW T2 reads B at time 5, T1 writes B at time 7
T1:R(B) → T2:W(B)   RW T1 reads B at time 6, T2 writes B at time 8
T1:W(B) → T2:W(B)   WW T1 writes B at time 7, T2 writes B at time 8

Total: 6 conflicts found.

Step 2: Build the Conflict Graph (Precedence Graph)

For each conflict, draw an arrow from the transaction that went first to the one that went second. The arrow means "this transaction MUST come before that one in any serial ordering."

Conflict Who goes first? Edge
T1:R(A) → T2:W(A) T1 acted first T1 → T2
T2:R(A) → T1:W(A) T2 acted first T2 → T1
T1:W(A) → T2:W(A) T1 acted first T1 → T2 (duplicate)
T2:R(B) → T1:W(B) T2 acted first T2 → T1 (duplicate)
T1:R(B) → T2:W(B) T1 acted first T1 → T2 (duplicate)
T1:W(B) → T2:W(B) T1 acted first T1 → T2 (duplicate)

Unique edges: T1 → T2 and T2 → T1

Step 3: Check for a Cycle

Read the graph:

  • T1 → T2 means "T1 must come before T2"
  • T2 → T1 means "T2 must come before T1"

Contradiction!

T1 must be before T2, AND T2 must be before T1. That's impossible — you can't have both. This is a cycle: T1 → T2 → T1 → T2 → ... forever.

Cycle found → NOT conflict serializable.

There is no serial ordering (neither "T1 then T2" nor "T2 then T1") that would produce the same result as this interleaved schedule. The schedule is unsafe.

4. For Comparison: A Schedule That IS Serializable

Time → 123456
T1 R(A)W(A)R(B)W(B)
T2 R(A)W(A)

No Cycle Here

Conflicts on A: T1:W(A) → T2:R(A) and T1:W(A) → T2:W(A). Both give edge T1 → T2.

All arrows point one direction: T1 → T2. No arrow goes back from T2 to T1.

No cycle → conflict serializable. This schedule is equivalent to running T1 first, then T2. Safe!

5. The Recipe (Use This on the Exam)

  1. List all operations in time order
    Write out the full interleaved schedule so you can see what happens when.
  2. Find all conflicts
    Check every pair of operations from different transactions on the same item. If at least one is a Write, it's a conflict. (RR is never a conflict!)
  3. Draw edges
    For each conflict, draw an arrow from the transaction that acted first to the one that acted second. Skip duplicates.
  4. Check for cycles
    If you can follow arrows and get back to where you started, there's a cycle → NOT serializable. If all arrows go one direction → serializable (equivalent to the topological order).

6. Why Does a Cycle Mean "Not Serializable"?

A serial schedule means one transaction runs completely before the other. There are only two options:

  • Option 1: T1 runs first, then T2
  • Option 2: T2 runs first, then T1

The edge T1 → T2 says "in any equivalent serial schedule, T1 must come first." The edge T2 → T1 says "T2 must come first." You can't satisfy both constraints at the same time.

Think of it like seating two people at a dinner table where Person A insists on sitting to the left of Person B, and Person B insists on sitting to the left of Person A. With only two chairs in a line, it's impossible to satisfy both. That's a cycle.

With 3+ transactions, cycles can be longer: T1 → T2 → T3 → T1. Same logic — you can't order them in a line where everyone's constraint is satisfied.

Key Takeaways

  • A conflict = two operations from different transactions, same item, at least one Write
  • The conflict graph has one node per transaction, and edges showing "must come before"
  • No cycle = serializable = safe (equivalent to some serial order)
  • Cycle = not serializable = the interleaved execution could produce results that no serial execution would
  • RR is never a conflict. Only RW, WR, and WW count.
