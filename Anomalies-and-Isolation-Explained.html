The Big Picture — Why Anomalies Exist

When multiple transactions run concurrently, their operations can interleave in ways that produce incorrect results. These incorrect results are called anomalies.

A shared Google Doc where two people edit the same paragraph at the same time. Without coordination, one person's changes can silently overwrite the other's.

There are four major anomalies. Each one is a different way that interleaving can go wrong:

The Four Anomalies at a Glance

AnomalyWhat Goes WrongOne-Line Summary
Dirty ReadReading uncommitted dataYou read a draft that gets thrown away
Lost UpdateTwo writes clobber each otherYour edit silently disappears
Unrepeatable ReadSame row, different value on re-readThe number changes while you're looking at it
Phantom ReadSame query, different rows on re-readNew rows magically appear mid-transaction

1. Dirty Read

Definition: Transaction Tx reads data that was written by Transaction Ty, but Ty has not yet committed. If Ty later aborts, Tx has used data that never officially existed.

Real-World Scenario

You're checking your bank balance. Meanwhile, a transfer of $500 into your account is being processed but hasn't finalized yet. You see $1,500 (including the $500). Then the transfer fails and is rolled back. Your actual balance is $1,000, but you already made a spending decision based on $1,500.

Step-by-Step Timeline

Step T1 (Balance Check) T2 (Transfer) Balance of Account A
1 — BEGIN $1,000
2 — UPDATE A SET bal = 1500 $1,500 (uncommitted)
3 READ A → sees $1,500 — DIRTY READ!
4 Makes decision based on $1,500 —
5 — ABORT (transfer fails) $1,000 (rolled back)

Why It's Dangerous

T1 used data that never existed in any committed state. If T1 already acted on this information (e.g., approved a loan, sent a confirmation), those actions are based on a lie. The data T1 read was a ghost — it was written and then erased.

You read someone's unsent draft email, then they delete the draft. You just acted on a message that was never actually sent.

2. Lost Update

Definition: Two transactions both read the same value, compute new values based on it, and write back. The second write overwrites the first, and T1's update is silently lost.

Real-World Scenario

A concert has 100 tickets left. Two people try to buy 1 ticket each at the same time. Both read "100 tickets left," both compute 100 − 1 = 99, and both write 99. Result: only 1 ticket was subtracted instead of 2. One sale was lost.

Step-by-Step Timeline

Step T1 (Buyer Alice) T2 (Buyer Bob) Tickets Remaining
1 READ tickets → 100 — 100
2 — READ tickets → 100 100
3 WRITE tickets = 99 — 99
4 — WRITE tickets = 99 99 (should be 98!)
5 COMMIT COMMIT Alice's update is LOST

Why It's Dangerous

No error is raised. No transaction aborts. Both commit successfully. But Alice's purchase silently vanished — the system sold 2 tickets but only decremented by 1. Over time, this leads to overselling, double-spending, or incorrect inventory.

Two people both edit cell B2 in a spreadsheet at the same time. The person who saves second unknowingly overwrites the first person's change. No warning is shown.

3. Unrepeatable Read

Definition: Transaction Tx reads the same row twice within one transaction and gets different values because another committed transaction Ty modified that row between the two reads.

Real-World Scenario

You're generating a financial report. You read a customer's balance at the start: $5,000. Halfway through the report, you read the same balance again to double-check. Now it says $3,000 because another transaction transferred $2,000 out and committed. Your report has contradictory numbers.

Step-by-Step Timeline

Step T1 (Report Generator) T2 (Transfer) Balance of Account A
1 READ A → $5,000 — $5,000
2 — UPDATE A SET bal = 3000 $3,000 (uncommitted)
3 — COMMIT $3,000 (committed)
4 READ A → $3,000 — UNREPEATABLE!

Why It's Dangerous

T1 sees two different values for the exact same row within a single transaction. The report now contains $5,000 on page 1 and $3,000 on page 3 for the same account. The data T2 wrote is perfectly valid (it committed!), but T1's view of the world is inconsistent.

Key difference from dirty read: In an unrepeatable read, the data T1 reads the second time is committed and valid. The problem isn't reading garbage — it's that T1's snapshot of reality changed mid-transaction.

4. Phantom Read

Definition: Transaction Tx executes the same range query twice and gets a different set of rows because another transaction Ty inserted or deleted rows matching the query's predicate between the two executions.

Real-World Scenario

You query "all employees in department 'Engineering'" and get 50 rows. While you're processing the list, HR hires 3 new engineers and commits. You re-run the same query and now get 53 rows. Three "phantom" rows appeared out of nowhere.

Step-by-Step Timeline

Step T1 (Report: Engineering headcount) T2 (HR: New hires) Engineering Rows
1 SELECT * WHERE dept='Eng' → 50 rows — 50
2 Processing the 50 rows... INSERT 3 new engineers 53 (uncommitted)
3 — COMMIT 53 (committed)
4 SELECT * WHERE dept='Eng' → 53 rows — 3 PHANTOMS!

Why It's Dangerous

The report header says "50 engineers" but the detail section lists 53. Or worse: a sum computed over 50 rows doesn't match a count computed over 53 rows. The set of rows changed under T1's feet — not the values of existing rows, but which rows exist at all.

Key difference from unrepeatable read: Unrepeatable read = same row, different value. Phantom read = same query, different rows. Unrepeatable reads involve UPDATE; phantom reads involve INSERT or DELETE.

Isolation Levels — The Defense System

SQL defines four isolation levels, each preventing progressively more anomalies. Higher isolation = safer but slower (more locking/blocking). The DBA chooses the right tradeoff for the workload.

Isolation Level Dirty Read Lost Update Unrepeatable Read Phantom Read
Read Uncommitted Possible Possible Possible Possible
Read Committed Prevented Possible Possible Possible
Repeatable Read Prevented Prevented Prevented Possible
Serializable Prevented Prevented Prevented Prevented

How Each Level Works

Read Uncommitted — No protection at all. Transactions can read anything, even uncommitted writes. Almost never used in practice.

Read Committed — You can only read data that has been committed. This prevents dirty reads (no reading drafts). But if another transaction commits an update between your two reads, you'll see the new value — so unrepeatable reads and phantoms are still possible.

Repeatable Read — Once you read a row, its value is "locked in" for the rest of your transaction. No one can change it until you're done. This prevents dirty reads and unrepeatable reads. But new rows can still be inserted into a range you queried — so phantom reads are still possible.

Serializable — The strongest level. Your transaction behaves as if it ran completely alone. All anomalies are prevented. Typically implemented using strict 2PL with predicate/range locks or MVCC with serializable snapshot isolation.

Read Uncommitted is an open office with no doors. Read Committed adds doors that lock while someone is writing. Repeatable Read adds locks on the rows you've already read. Serializable locks entire ranges so no one can even add new rows in your query's territory.

Why Not Always Use Serializable?

Because it's expensive. More locking = more transactions waiting = lower throughput. The tradeoff:

Higher Isolation

Safer. Fewer anomalies. Easier to reason about correctness.

Higher Isolation

Slower. More contention. Lower throughput. More deadlocks.

Most production databases default to Read Committed (PostgreSQL, Oracle, SQL Server) because it's a good balance. Applications handle the remaining anomalies in application logic or use explicit locks for critical sections.

Strict Serializability — The Ultimate Guarantee

Now we can understand the concept from Q21 on the practice midterm. There are actually two levels of "serializable" guarantees:

Conflict Serializability vs. Strict Serializability

Conflict Serializability (what we learned in Week 4)

A schedule is conflict serializable if it is equivalent to some serial order. The database checks this using the conflict graph (precedence graph). If the graph is acyclic (a DAG), the schedule is safe.

But it doesn't care about real-world time. If transaction X finishes at 2:00 PM and transaction Y starts at 3:00 PM, conflict serializability might still produce a serial order where Y comes before X. Logically correct, but confusing to humans.

Strict Serializability (the stronger version)

Strict serializability = conflict serializability + real-time ordering.

The additional rule: If X finishes before Y starts (they don't overlap in real time), then X must appear before Y in the equivalent serial order.

This means the database's behavior matches what you'd expect from watching the clock.

Why Does the Difference Matter?

Consider this scenario:

Real Time Transaction X Transaction Y
2:00 BEGIN —
2:01 WRITE A = 10 —
2:02 COMMIT —
3:00 — BEGIN
3:01 — READ A → ???
3:02 — COMMIT

Conflict Serializable

Y could read A = 10 or the old value of A (if equivalent to serial order Y, X). Both are technically "correct" — they match some serial schedule.

Strictly Serializable

Y must read A = 10. Because X committed at 2:02 and Y started at 3:00, X must precede Y. Any other result would violate real-time ordering.

In Plain English

Conflict serializable says: "The end result matches some serial order."

Strictly serializable says: "The end result matches the serial order you'd expect from looking at the clock."

Strict serializability is what users intuitively assume databases do. If I deposit money at 2 PM and you check at 3 PM, you should see my deposit. Conflict serializability alone doesn't guarantee this — strict serializability does.

The Hierarchy

Serial Schedules (run one at a time)
   ⊂ Strictly Serializable (respects real-time order)
      ⊂ Conflict Serializable (equivalent to some serial order)
         ⊂ Serializable (result-equivalent to some serial order)

Every serial schedule is strictly serializable. Every strictly serializable schedule is conflict serializable. But not the other way around. Each level relaxes the constraint slightly, allowing more concurrency but giving a weaker guarantee.

The midterm tests whether you understand the additional constraint that strict serializability adds: real-time ordering of non-concurrent transactions. Conflict serializability guarantees correctness. Strict serializability guarantees correctness and matches human intuition about time.
