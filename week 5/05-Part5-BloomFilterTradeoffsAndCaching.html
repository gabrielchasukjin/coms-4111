Bloom Filter Trade-offs and Caching

Optimizing false positive rates, memory trade-offs, and introduction to cache eviction policies

1. False Positive Probability and Optimal k

Continuing from Part 4, the size of k (the number of hash functions) presents a fundamental trade-off:

Higher k Helps

More hash functions means more bit positions must all be 1 for a false positive to occur. Each additional hash function that maps to a 0 catches the impostor. This reduces false positives.

Higher k Hurts

Each inserted element sets k bits to 1. More hash functions "deplete" the available 0 slots faster. Eventually the array fills up with 1s, and everything looks like a member.

Optimal k: The value of k that minimizes the false positive rate is k = (m/n) × ln(2), where m is the number of bits and n is the number of elements. For example, with m/n = 8 bits per member, the optimal k = 8 × ln(2) ≈ 5.45.

The false positive rate as a function of k forms a U-shaped curve: too few hash functions means insufficient filtering, too many means the array fills up too fast. The minimum sits right at k = (m/n) × ln(2).

Bloom Filter Calculator

The lecture references an online tool at hur.st/bloomfilter/ that lets you explore how the parameters interact. For example, with n = 4,000 items and a target false positive rate of p = 0.0000001 (1 in ~10 million), the calculator computes:

  • m = 134,191 bits (about 16.4 KiB)
  • k = 23 hash functions

2. Memory Space vs. Speed Trade-off

A larger Bloom filter reduces false positives, which means fewer unnecessary disk reads, lower latency, and higher throughput. But a larger Bloom filter also takes up more memory — and memory that goes to the Bloom filter cannot be used for caching index entries. Less index caching means more disk accesses for keys that do exist, which increases latency.

There is a tension: the Bloom filter and the index cache compete for the same finite memory. Making the Bloom filter bigger helps negative lookups but hurts positive lookups. The optimal split depends on the workload.

Worked Example: Space vs. Speed

Consider a database with 2GB of memory for caching, 100GB of flash storage, supporting 1 billion entries (~100B each). Flash access costs 100μs, memory access costs 100ns. Suppose 60% of requests are for objects that do not exist.

Scenario 1: 600MB Bloom Filter

False positive rate: 9%

Remaining DRAM for index cache: 1.4GB

Index cache hit rate: 90%

Scenario 2: 1.1GB Bloom Filter

False positive rate: 1%

Remaining DRAM for index cache: 0.9GB

Index cache hit rate: 70%

Average Latency Formula

avg latency = P(exists) × [P(cached) × 100ns + P(not cached) × 100μs] + P(doesn't exist) × [P(true negative) × 100ns + P(false positive) × 100μs]

Scenario 1: 0.4 × (0.9 × 100ns + 0.1 × 100μs) + 0.6 × (0.91 × 100ns + 0.09 × 100μs) = 9.49μs

Scenario 2: 0.4 × (0.7 × 100ns + 0.3 × 100μs) + 0.6 × (0.99 × 100ns + 0.01 × 100μs) = 12.64μs

Surprise: The Smaller Bloom Filter Wins!

Despite having a 9% false positive rate (vs. 1%), Scenario 1 achieves lower average latency (9.49μs vs. 12.64μs). The extra 800MB of index cache improves hit rates from 70% to 90%, which more than compensates for the higher false positive rate. This illustrates why the trade-off must be evaluated holistically — the Bloom filter does not exist in isolation.

3. Other Issues with Bloom Filters

Bloom filters have several practical limitations:

  • Depletion over time: As more elements are added, more bits become 1, and the false positive rate increases. Eventually the filter becomes useless if too many elements are inserted.
  • Must estimate n in advance: You need to know (or estimate) the total number of unique entries that will ever be inserted, since m and k are chosen based on n. Underestimating n leads to a depleted filter with high false positive rates.
  • No deletes: Standard Bloom filters do not support removing elements. Why? Because setting a bit back to 0 might affect other elements that also hash to that position. Clearing one element's bit could create false negatives for other elements.

Variants and Improvements

Research has produced many improved variants: counting Bloom filters (replace bits with counters to support deletes), cuckoo filters (support deletes and can be more space-efficient), learned Bloom filters (use machine learning to reduce false positives), and elastic Bloom filters (grow dynamically). This remains a hot research area in systems and databases.

4. Introduction to Caching

The lecture transitions from Bloom filters to a closely related topic: caching. Caches appear at every level of computer systems to reduce work and transport duplication:

  • Databases cache frequently accessed keys in memory
  • CPUs use memory (DRAM) as a cache for disk
  • Content delivery networks (CDNs) are caches for origin servers
  • The file system has a page cache
  • Browsers cache web pages and assets

In big data systems, dedicated key-value cache servers sit between the application and the database. All data is stored purely in memory (no disk), and the application checks the cache before querying the database. Examples include Memcached, Redis, and Amazon ElastiCache.

The lecture references a paper by Maggs and Sitaraman on "Algorithmic Nuggets in Content Delivery," which describes how Akamai (one of the world's largest CDNs) uses sophisticated caching algorithms at global scale — a real-world application of the principles covered here.

5. Important Questions for Caching

Designing a caching system requires answering three fundamental questions:

1 Assignment: Where do you put the data?

In memory? How is it organized? The goal is to optimize read speed — finding a cached item should be fast (typically O(1) via hash tables).

2 Consistency: What happens when data is updated or deleted?

Two approaches:

  • Passive: cache entries have a TTL (time-to-live) and simply expire after a set duration
  • Active: cache entries are explicitly invalidated when the underlying data changes
3 Eviction: Who do you kick out?

Cache space is expensive (that's why we have a cache in the first place — if memory were infinite, we'd store everything). When the cache is full and a new item arrives, which existing item gets removed? This is the eviction policy.

6. Standard Eviction Policy Model

Model Assumptions:
  • Cache has a fixed size C (number of slots)
  • Each item has the same size (1 slot) and a unique key
  • The cache starts full
  • On a hit (requested item is in cache): read directly from cache
  • On a miss (requested item is not in cache): fetch from the backing store (e.g., database), insert into cache, and evict one existing item to make room

Metric for Success: Cache Hit Rate

The overall goal is usually to minimize average latency, which in practice is equivalent to maximizing the cache hit rate — the percentage of read requests that find their object already in the cache.

Why Small Hit Rate Differences Matter Enormously

Consider: average latency to flash database = 5ms, average latency to in-memory cache = 100μs.

At 98% hit rate: 0.98 × 100μs + 0.02 × 5000μs = 198μs

At 99% hit rate: 0.99 × 100μs + 0.01 × 5000μs = 149μs

A mere 1% improvement in cache hit rate reduces average latency by 25%. This is because the backing store is so much slower that even a tiny reduction in misses has an outsized impact on the average.

7. Common Eviction Policies

The lecture introduces five eviction strategies, which will be explored in detail in subsequent parts:

Policy Strategy Key Idea
FIFO First In, First Out Evict the item that has been in the cache the longest. Simple queue — no need to track access patterns.
LRU Least Recently Used Evict the item that was accessed least recently. Assumes recently used items are likely to be used again soon.
CLOCK Approximates LRU A practical approximation of LRU using a circular buffer and reference bits. Cheaper to implement than true LRU.
LFU Least Frequently Used Evict the item with the fewest total accesses. Favors items that are accessed often, regardless of recency.
OPT Belady's Algorithm Evict the item that will not be needed for the longest time in the future. Optimal but requires knowing future requests — used as a theoretical benchmark.
Each policy makes different assumptions about access patterns. FIFO ignores usage entirely. LRU bets on temporal locality (recently used = likely used again). LFU bets on frequency (popular items stay popular). OPT is the theoretical best but impossible to implement in practice — it serves as the upper bound against which real policies are measured.

Key Takeaways from Part 5

  • ✓ Optimal k for a Bloom filter is k = (m/n) × ln(2) — a U-shaped trade-off between too few and too many hash functions
  • ✓ Bloom filters compete with index caches for finite memory — a bigger Bloom filter helps negative lookups but hurts positive lookups
  • ✓ The smaller Bloom filter can win when the workload benefits more from index caching than from reduced false positives
  • ✓ Bloom filters cannot delete elements, can get depleted, and require estimating n in advance
  • ✓ Caches appear at every layer of systems: CPU, file system, database, CDN, application
  • ✓ Three key caching questions: assignment (where), consistency (staleness), eviction (who to remove)
  • ✓ A 1% cache hit rate improvement can yield a 25% latency reduction due to the enormous speed gap between cache and backing store
  • ✓ Five eviction policies: FIFO, LRU, CLOCK, LFU, and OPT (Belady's optimal, theoretical benchmark)
