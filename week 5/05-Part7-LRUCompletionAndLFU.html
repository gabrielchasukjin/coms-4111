LRU Completion, LRU Approximations, and LFU

Completing the LRU trace, practical approximations, and frequency-based eviction

1. LRU Worked Example (Complete)

Continuing from Part 6, here is the full LRU trace for the sequence 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 with cache size 4. The cache is ordered by recency — leftmost is least recently used (next to be evicted).

Request Result Action Cache (LRU → MRU)
1 Miss Insert 1 [1]
2 Miss Insert 2 [1, 2]
3 Miss Insert 3 [1, 2, 3]
4 Miss Insert 4 (full) [1, 2, 3, 4]
1 Hit Move 1 to MRU end [2, 3, 4, 1]
2 Hit Move 2 to MRU end [3, 4, 1, 2]
5 Miss Evict 3 (LRU), insert 5 [4, 1, 2, 5]
1 Hit Move 1 to MRU end [4, 2, 5, 1]
2 Hit Move 2 to MRU end [4, 5, 1, 2]
3 Miss Evict 4 (LRU), insert 3 [5, 1, 2, 3]
4 Miss Evict 5 (LRU), insert 4 [1, 2, 3, 4]
5 Miss Evict 1 (LRU), insert 5 [2, 3, 4, 5]

Result: 8 misses, 4 hits out of 12 requests

Hit rate: 4/12 = 33.3% — better than FIFO's 16.7%!

The key difference was at request 5 (item "1"): FIFO left item 1 at the front of the eviction queue (oldest inserted), so it was evicted when 5 arrived. LRU moved item 1 to the back (most recently used), so it survived. This gave LRU 2 extra hits (requests 8 and 9). However, LRU is still not optimal…

Notice the critical mechanic: on a cache hit, LRU rearranges the cache order even though no item is inserted or evicted. This is what makes LRU more expensive to implement than FIFO — every single access (not just misses) requires updating the data structure.

2. LRU Approximations

True LRU is expensive to implement: every read requires updating a linked list to move the accessed item to the most-recent position. Maintaining a full ranking of all items at every access can become a bottleneck. In practice, many caches use approximations of LRU that are cheaper but nearly as effective.

Sampled LRU

How It Works

  • Instead of maintaining a full ordering (linked list) of all items, simply record the last access timestamp for each item when it is accessed
  • At eviction time, randomly sample N items from the cache and evict the one with the oldest access time
  • When N is large enough (e.g., 50 or 100), this approximates true LRU very well

This is the approach used by Redis. It avoids the overhead of maintaining a sorted list on every access — instead, you only do work at eviction time, and even then it's just sampling rather than scanning the entire cache.

CLOCK Algorithm

CLOCK: An LRU approximation that maintains just 1 bit per item in the cache (a "reference bit" that can be 0 or 1). It uses a circular buffer with a "clock hand" pointer.
1 On access: set the item's bit to 1

This marks the item as "recently used." This is much cheaper than moving a node in a linked list.

2 On eviction: sweep the clock hand

The hand moves around the circular buffer. For each item it passes: if the bit is 1, set it to 0 and move on (give it a "second chance"). If the bit is 0, evict that item — it hasn't been accessed since the last sweep.

CLOCK was originally designed for operating system page cache management. It captures the essence of LRU (recently accessed items survive) with minimal overhead (1 bit per item, no list reordering).

3. LFU (Least Frequently Used)

LFU Policy: Evict the item that has been accessed the fewest total times (lowest frequency count). The idea: items that have been accessed many times are likely to be accessed again in the future.

Advantages

  • Works well for workloads with a relatively static set of popular items
  • Popular items accumulate high counts and are protected from eviction

Disadvantages

  • Does not adapt well when new items become popular — old items with high historical counts block new popular items from staying in cache
  • Expensive to implement: must track frequency for all keys and find the minimum
  • To fully implement, also need to track frequencies of evicted items (so they retain their count if re-inserted)

The key distinction from LRU: LRU cares about when you were last accessed (recency), while LFU cares about how many times you've been accessed (frequency). These can lead to very different eviction decisions.

LFU Worked Example (Beginning)

Same sequence: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 with cache size 4. Now we track access counts.

Request Result Cache Counts
1 Miss [1] 1:1
2 Miss [1, 2] 1:1, 2:1
3 Miss [1, 2, 3] 1:1, 2:1, 3:1
4 Miss [1, 2, 3, 4] 1:1, 2:1, 3:1, 4:1
1 Hit [1, 2, 3, 4] 1:2, 2:1, 3:1, 4:1

After these 5 steps, item 1 has count 2 while items 2, 3, 4 each have count 1. The next request (item 2) will be a hit, raising its count to 2. Then when item 5 arrives, LFU must evict the item with the lowest frequency — either 3 or 4 (both at count 1). The full LFU trace continues in Part 8.

Notice how LFU makes a fundamentally different decision than both FIFO and LRU. When item 5 arrives, FIFO evicts item 1 (oldest inserted), LRU evicts item 3 (least recently used), and LFU evicts item 3 or 4 (lowest frequency). Each policy "bets" on a different signal for predicting future access.

Think of three different strategies for deciding which clothes to donate from a full closet. FIFO: donate whatever you bought earliest. LRU: donate whatever you wore least recently. LFU: donate whatever you've worn the fewest total times. Each captures a different aspect of "usefulness," and the best choice depends on your wearing habits.

Key Takeaways from Part 7

  • ✓ LRU achieves 8 misses on the reference sequence vs. FIFO's 10 — a 33.3% hit rate vs. 16.7%
  • ✓ LRU's advantage: on a hit, items are moved to the MRU end, protecting frequently re-accessed items from eviction
  • ✓ LRU is expensive: requires reordering on every access, not just on misses
  • ✓ Sampled LRU: record timestamps, sample N items at eviction time, evict the oldest — used by Redis
  • ✓ CLOCK: 1 reference bit per item + circular sweep — originally used for OS page caching
  • ✓ LFU evicts the item with the lowest total access count — good for stable popularity, bad for shifting trends
  • ✓ LFU is also expensive: must track frequencies for all keys (including evicted ones) and find the minimum
  • ✓ Each policy bets on a different signal: insertion time (FIFO), recency (LRU), or frequency (LFU)
