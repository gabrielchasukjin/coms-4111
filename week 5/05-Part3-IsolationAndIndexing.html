Isolation Levels and Indexing

From real-world consistency trade-offs to the data structures that make databases fast

A. Isolation Levels Beyond Serializability

In an ideal world, every database would give us serializability — the guarantee that concurrent transactions behave as if they ran one at a time. But we do not live in an ideal world. Serializability is unfortunately rare in production systems. Some databases even claim to be serializable but actually are not (Oracle's "serializable" mode, for example, historically provided something weaker). Why? Because full serializability is expensive. It requires the database to restrict concurrency, which slows things down. So the industry settled on a spectrum of isolation levels — each making different trade-offs between correctness and performance.

1. The Isolation Level Spectrum

The SQL standard defines four isolation levels, ordered from weakest to strongest. To understand them, we first need to name the anomalies they protect against:

Dirty Read: Transaction T1 modifies a row. Transaction T2 reads that row before T1 commits. If T1 then aborts, T2 has read data that never officially existed.

Lost Update: T1 reads a value, T2 reads the same value, both modify and write it back. One update overwrites the other silently.

Unrepeatable Read (Non-repeatable Read): T1 reads a row. T2 then modifies or deletes that row and commits. If T1 reads the same row again, it gets a different result.

Phantom Read: T1 reads a set of rows satisfying some condition. T2 inserts a new row that satisfies that condition and commits. If T1 re-executes the same query, it sees a "phantom" row that was not there before.

Read Committed

This is the most common isolation level in SQL OLTP systems (it is the default in PostgreSQL, Oracle, and SQL Server). The guarantee is simple: any data you read was committed at the moment you read it. This prevents dirty reads — you will never see uncommitted data from another transaction. However, Read Committed still allows lost updates, unrepeatable reads, and phantom reads.

Think of Read Committed like reading a newspaper. You are guaranteed that every article you read was published (committed). But if you read the front page, go get coffee, and come back, the front page might have a different headline now — the newspaper was updated between your reads. That is an unrepeatable read, and Read Committed allows it.

Repeatable Reads

This level is stronger than Read Committed. Once your transaction reads a row, it is guaranteed to see the same value if it reads that row again — even if another transaction modifies and commits it in between. This eliminates unrepeatable reads. However, Repeatable Reads still allows lost updates and phantom reads. New rows can appear in your query results because the lock is on specific rows you have read, not on the range of possible rows matching your query condition.

Most production OLTP databases run at Read Committed, not Serializable. This means that as application developers, we must be aware that anomalies like lost updates and phantom reads can and do happen. Designing correct applications requires understanding exactly what guarantees your database provides — and which ones it does not.

2. Strict Serializability — The Gold Standard

Even if a database provides full serializability (conflict serializability, as we discussed earlier), there are subtle limitations that may surprise you:

Limitations of Plain Serializability

  • No constraints on equivalent order: Serializability guarantees that the final result is equivalent to some serial order, but it says nothing about which serial order. The database is free to pick any equivalent ordering.
  • No relationship to wall-clock time: There is no requirement that the equivalent serial order respects when transactions were actually submitted. A transaction submitted later could appear "before" an earlier one in the equivalent serial order.
  • Non-determinism across replicas: Two identical databases given the exact same transactions in the exact same order may produce different results — because they could pick different (but both valid) equivalent serial orders.

This is where strict serializability comes in. It adds a crucial constraint on top of conflict serializability.

Strict Serializability: If transaction Y starts after transaction X commits (i.e., X and Y are not concurrent), then:
  1. The final state is equivalent to some serial order (just like regular serializability), AND
  2. X must appear before Y in that serial order.

In other words, for non-concurrent transactions, the equivalent serial order must respect real-time ordering. The database is only free to reorder transactions that were actually concurrent.
Imagine two people taking turns at a single ATM. Plain serializability says the final bank balance must look as if the two people used the ATM one at a time — but it does not guarantee that the person who went first in real life is reflected as going first. Strict serializability adds: if Alice clearly finished before Bob walked up, then the result must reflect Alice going first. The only flexibility is when Alice and Bob are both pressing buttons at the same time (true concurrency).

Why does this matter?

Consider a distributed system with two database replicas. Under plain serializability, both replicas might independently choose different equivalent serial orders, leading to divergent states — even though they received the same transactions. Strict serializability eliminates this problem for non-concurrent transactions, because the real-time ordering constrains the allowed serial order. This is essential for systems where users expect "if I did X before Y, the database should reflect that."

The 2PL Connection

Two-Phase Locking (2PL), which we studied earlier, naturally produces strictly serializable schedules. Because 2PL forces transactions to acquire all their locks before releasing any, it inherently respects real-time ordering: a transaction that starts after another commits will always see the committed transaction's effects. This is one of the reasons 2PL was the dominant concurrency control method for decades, despite its performance costs.

3. MVCC — Multi-Version Concurrency Control

Modern databases overwhelmingly use a technique called Multi-Version Concurrency Control (MVCC) rather than traditional locking. The core idea is elegant: instead of blocking readers when a writer is active (as 2PL does), the database maintains multiple versions of each data item. Readers can access an older, consistent version while a writer creates a new version. This dramatically increases parallelism.

MVCC: A concurrency control method where the DBMS maintains multiple physical versions of a single logical object. When a transaction writes to an object, the DBMS creates a new version. When a transaction reads an object, it reads the newest version that existed when the transaction started.

MVCC is the most popular transaction management scheme in modern DBMS. It is used by PostgreSQL, MySQL/InnoDB, Oracle, SQL Server (in snapshot isolation mode), and virtually every major database engine. The research paper "An Empirical Evaluation of In-Memory Multi-Version Concurrency Control" (Wu et al., VLDB 2017) provides an excellent empirical comparison of MVCC variants. The key insight from the paper is that while MVCC increases parallelism, the design choices in how versions are stored, garbage-collected, and indexed have profound performance implications.

Think of MVCC like Google Docs' version history. Every edit creates a new version, but other users can still read the previous version without being blocked. Eventually, old versions get cleaned up (garbage collected), but at any moment, multiple consistent snapshots of the document exist simultaneously.

B. Indexing

We now shift from transaction management to a question that is equally fundamental: how do we find data quickly? A database might store millions or billions of rows. If every query required scanning every single row, databases would be unusably slow. Indexing is the mechanism that makes fast data access possible.

4. The Basic Idea

Consider a library with thousands of books. Without any organization system, finding a specific book would require walking through every shelf and examining every spine. An author catalog solves this problem: it is a small, organized lookup structure that tells you exactly where to find any book by its author. A database index serves exactly the same purpose — it is a smaller, organized structure that tells the database exactly where to find rows matching a given value.
Search Key: An attribute (or set of attributes) used to look up records in a file. Note: a search key is not the same as a primary key — any attribute can serve as a search key.

Index: A data structure consisting of index entries of the form (search-key value, pointer to record). The index is typically much smaller than the original data — often 100 times smaller or more.

Why is the index so much smaller?

Each index entry stores only two things: the search key value and a pointer. The pointer is just an address — its size depends on how much data you have. For example, to address any byte within 1 GB of data, you need 30 bits (since 230 = 1 GB). Compare that to a full record which might contain dozens of attributes totaling hundreds of bytes. The index strips away everything except the key and the pointer, making it compact enough to fit entirely in memory in many cases. And when the index fits in memory, lookups become extraordinarily fast.

5. Index Evaluation Metrics

Not all indices are created equal. When choosing an index structure, we evaluate it along several dimensions:

1 Access types supported

Can the index handle point queries (find records with a specific key value, e.g., "find instructor with ID 12345") and range queries (find records with keys in a range, e.g., "find all instructors with salary between 80000 and 100000")?

2 Access time

How long does it take to find a particular record or set of records using the index?

3 Insertion time

How long does it take to insert a new record and update the index accordingly?

4 Deletion time

How long does it take to delete a record and update the index?

5 Space overhead

How much additional storage space does the index require? There is always a trade-off: more index structure means faster reads but more storage and slower writes.

6. Ordered Indices

An ordered index stores index entries sorted on the search key value. This is the most common type of index and the one we will focus on. Within ordered indices, there is an important distinction:

Primary Index (Clustering Index)

The search key of the index specifies the same order as the sequential (physical) order of the file. For example, if the instructor file is physically sorted by ID, then an index on ID is a primary index. Note: the search key of a primary index is usually, but not necessarily, the primary key of the relation.

Secondary Index (Non-clustering Index)

The search key specifies a different order from the sequential order of the file. For example, if the instructor file is sorted by ID, but we build an index on salary, that is a secondary index. The salary values in the index are sorted, but the actual records on disk are not in salary order.

A file can have at most one primary index (because the data can only be physically sorted in one order), but it can have many secondary indices (one for each attribute or combination of attributes we want to search on quickly).

7. Dense Index Files

In a dense index, there is an index entry for every search-key value in the file. If we have an instructor relation with 100 distinct ID values, our dense index on ID will have exactly 100 entries, each pointing to the corresponding record.

Example: Dense index on the instructor relation

Imagine our instructor relation has records for instructors with IDs 10101, 12121, 15151, 22222, 32343, 33456, 45565, 58583, 76543, 76766, 83821, and 98345. A dense index on the ID attribute would contain twelve entries, sorted by ID. Each entry stores the ID value and a pointer to the disk block (and offset within the block) where that instructor's record lives. To find instructor 32343, we search the index (which is small and likely in memory), find the entry for 32343, follow the pointer, and retrieve the full record in a single disk access.

Dense indices also work perfectly well with non-unique attributes. Consider a dense index on the dept_name attribute of the instructor relation. If there are 5 instructors in the "Physics" department, the index entry for "Physics" would point to the first of those records (or to a list/bucket of pointers to all of them).

8. Sparse Index Files

A sparse index contains entries for only some of the search-key values. Typically, one entry per disk block of the data file. This only makes sense when the data file is sequentially ordered on the search key — otherwise, how would you find values that are not in the index?

Sparse index lookup procedure: To find a record with search key value K, find the index entry with the largest key value that is less than or equal to K. Follow the pointer to that block, then search sequentially from there until you find K (or determine it does not exist).
A dense index is like a phone book with every person's name listed. A sparse index is like a phone book where only the first name on each page is listed in the table of contents. To find "Johnson," you look up the table of contents, find that page 47 starts with "Jenkins" and page 48 starts with "Jones." You go to page 47 and scan forward until you find "Johnson." You still find it — just with one extra step.

Dense vs. Sparse: The Trade-off

Dense Index

  • Entry for every search-key value
  • Faster lookups (direct pointer)
  • More space required
  • More maintenance on inserts/deletes
  • Can be used on any attribute

Sparse Index

  • Entry for only some search-key values
  • Slower lookups (requires sequential scan)
  • Less space required
  • Less maintenance overhead
  • Only works when data is sorted on the search key
The best trade-off often lies between the two extremes: a sparse index with one entry per disk block. This minimizes index size while ensuring that at most one block needs to be read to find any record (after the index lookup, you need to scan at most one block).

9. Secondary Index Example

Consider building a secondary index on the salary field of the instructor relation. The data file is sorted by ID (the primary key), not by salary. This means instructor records with similar salaries are scattered across different disk blocks.

How a secondary index works

The index entries are sorted by salary. Each entry contains a salary value and a pointer. But since multiple instructors can have the same salary, the pointer does not go directly to a single record. Instead, it points to a bucket — a block containing pointers to all records with that salary value. For example, if three instructors earn $95,000, the index entry for 95000 points to a bucket that contains three pointers, one for each of those instructor records (which may live in three different disk blocks since the file is not sorted by salary).

Why must secondary indices be dense?

Recall that sparse indices work by finding the nearest smaller key in the index and then scanning sequentially. But sequential scanning only works when the data is physically sorted by the search key. In a secondary index, the data is not sorted by the indexed attribute — so sequential scanning from a nearby point will not help you find the target record. You must have a direct pointer for every distinct search-key value. Therefore, secondary indices must always be dense.

10. Multilevel Index

So far, we have assumed the index fits in memory. But what happens when the index itself is too large for memory? If we have billions of records, even a compact index could be tens of gigabytes. Searching a disk-resident index sequentially would be slow.

The solution is beautifully recursive: treat the index itself as a sequential file and build another index on top of it.

1 Inner Index (Basic Index)

This is our original index file. It is ordered by search key and may be too large to fit in memory. Think of it as the "detailed" level.

2 Outer Index (Index of the Index)

This is a sparse index built on top of the inner index. Since the inner index is a sorted file, we can build a sparse index on it with one entry per block. The outer index is much smaller — small enough to fit in memory.

3 Additional Levels (if needed)

If the outer index is still too large, we can add yet another level on top. This generalizes to any number of levels, though in practice two or three levels suffice for very large datasets. (This recursive indexing idea eventually leads us to B+ trees, which we will study in detail.)

Imagine a massive encyclopedia. The inner index is like the detailed index at the back of the encyclopedia (possibly hundreds of pages itself). The outer index is like the index of the index: "A-D starts on index page 1, E-H starts on index page 47, ..." — a short guide that helps you find the right section of the detailed index quickly.

The maintenance cost

The downside of multilevel indices is that all levels must be updated on every insertion and deletion. When a new record is inserted, the inner index may need a new entry, which might cause its block structure to change, which in turn requires updating the outer index. This cascading update cost is one of the motivations for more sophisticated index structures like B+ trees, which handle insertions and deletions gracefully through splitting and merging of nodes.

11. Index Update: Insertion

When a new record is inserted into the database, the index must be updated to reflect the new data. The procedure differs depending on the index type:

Dense Index Insertion

If the new record's search-key value is not already in the index, a new index entry is inserted in the correct sorted position. If the search-key value already exists (non-unique key), the system adds the new record's pointer to the existing entry's pointer list or bucket.

Sparse Index Insertion

Since a sparse index only has entries for some values (typically one per block), an update is needed only if the insertion causes a new block to be created in the data file (e.g., when a block overflows and is split). In that case, the first search-key value in the new block is added to the index.

Multilevel insertion and deletion

For multilevel indices, insertion and deletion algorithms are straightforward extensions of the single-level case. An insertion into the inner index might cause a new block in the inner index, which triggers an insertion into the outer index, and so on up the levels. Deletion is the mirror image: removing an entry from the inner index might cause a block merge, triggering a deletion from the outer index. These cascading updates are manageable but add overhead, which is why efficient self-balancing structures (like B+ trees) are preferred in practice.

Key Takeaways

  • Serializability is rare in practice. Most OLTP databases default to Read Committed, which prevents dirty reads but allows lost updates, unrepeatable reads, and phantom reads. Knowing your database's isolation level is essential for building correct applications.
  • Strict serializability is the gold standard. It adds real-time ordering constraints on top of conflict serializability: non-concurrent transactions must appear in the serial order consistent with when they actually ran. 2PL naturally provides this guarantee.
  • MVCC dominates modern databases. By maintaining multiple versions of data, MVCC allows readers and writers to proceed concurrently without blocking each other, dramatically improving throughput.
  • Indices make databases fast. An index is a compact lookup structure of (search-key, pointer) pairs, typically 100x smaller than the data, often fitting entirely in memory.
  • Primary vs. secondary indices: A primary index matches the physical sort order of the data; a secondary index does not. A file can have at most one primary index but many secondary indices.
  • Dense vs. sparse indices: Dense indices have an entry for every key value (fast but large). Sparse indices skip some values (smaller but require sequential scan). Secondary indices must always be dense.
  • Multilevel indices solve the "index too large for memory" problem by recursively indexing the index itself. This idea generalizes into B+ trees.
