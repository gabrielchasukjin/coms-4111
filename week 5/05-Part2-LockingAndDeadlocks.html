Two-Phase Locking and Deadlocks

How databases use locks to enforce serializability — and what happens when they get stuck

1. The Idea: Use Locks for Concurrency Control

In the previous part, we established that conflict serializability is the gold standard for correct concurrent execution. We learned how to test whether a given schedule is conflict serializable by building a conflict graph and checking for cycles. But testing after the fact is not enough — we need a way to produce correct schedules in real time as transactions arrive and execute.

The answer is locking. The idea is beautifully simple: before a transaction can read or write a data item, it must first acquire a lock on that item. If the lock is held by another transaction in a conflicting mode, the requesting transaction must wait. This waiting is what prevents conflicting operations from interleaving incorrectly.

Think of a lock like a bathroom door lock. When someone is inside (holding the lock), you have to wait outside until they finish and unlock the door. The lock itself does not do the work — it simply coordinates access so two people are never inside at the same time.

But not all access is the same. Reading is inherently less dangerous than writing. Two people can both look at a painting in a museum at the same time without conflict. But if one person is repainting it, nobody else should be looking at or painting the same canvas. This distinction gives rise to two types of locks.

2. Strict Two-Phase Locking (Strict 2PL)

Strict Two-Phase Locking (Strict 2PL) is a concurrency control protocol that guarantees conflict serializability (if the schedule completes). It is conceptually straightforward and transparent to the user — the database manages locks automatically behind the scenes.

The Two Lock Types

Strict 2PL uses two types of locks:

S Lock (Shared)

Acquired before reading an object. Multiple transactions can hold an S lock on the same object simultaneously. This makes sense: reads do not conflict with reads.

Prevents other transactions from acquiring an X lock on the same object.

X Lock (Exclusive)

Acquired before writing an object. Only one transaction can hold an X lock at a time. No other transaction can hold any lock (S or X) on the same object.

Prevents other transactions from acquiring either S or X locks.

Lock Compatibility Matrix

The following table shows whether a new lock request is compatible with an existing lock held by another transaction. "Yes" means the request is granted immediately; "No" means the requesting transaction must wait.

Request \ Held No Lock S Lock (Shared) X Lock (Exclusive)
S Lock Yes Yes No — must wait
X Lock Yes No — must wait No — must wait
The key pattern: shared locks are compatible with other shared locks (readers can coexist), but exclusive locks are compatible with nothing (writers need total isolation on that object). This directly mirrors our understanding of conflicts — read-read is fine, but read-write and write-write are not.

The Strict 2PL Protocol Rules

Three Simple Rules

  1. Before reading object O, a transaction must acquire an S (shared) lock on O. The request is blocked if another transaction holds an X lock on O.
  2. Before writing object O, a transaction must acquire an X (exclusive) lock on O. The request is blocked if another transaction holds any lock (S or X) on O.
  3. All locks are released only when the transaction completes — that is, at COMMIT or ABORT time. Not before.

Rule 3 is what makes the protocol strict. The transaction holds onto every lock it acquires until the very end. This is the "two-phase" part: there is a growing phase where the transaction accumulates locks, followed by a single moment of release when all locks are dropped simultaneously at commit/abort.

3. The Picture of Two-Phase Locking

To understand the name "two-phase locking," it helps to visualize how the number of locks held by a transaction changes over time. There are actually two variants worth distinguishing.

Basic 2PL

Growing phase: The transaction acquires locks as needed. It may acquire locks at different times as it encounters new objects to read or write.

Shrinking phase: Once the transaction releases any lock, it can never acquire a new lock again. It can only continue releasing locks.

The critical rule: you cannot request new locks after releasing any lock.

Strict 2PL

Growing phase: Same as basic 2PL — locks are acquired as needed throughout execution.

Shrinking phase: All locks are released at once, right at COMMIT or ABORT. There is no gradual release.

The shrinking phase is a single instant: release everything at the end.

Imagine you are at a buffet and you can only carry food with your hands (no plates). Basic 2PL says: once you put down any food item, you cannot pick up anything new. Strict 2PL says: you carry everything until you sit down at your table, and only then do you put everything down at once. Strict 2PL is the simpler, more conservative version — and it is the one databases actually use.
Strict 2PL is strictly stronger than basic 2PL. Every strict 2PL schedule is also a basic 2PL schedule, but not vice versa. Strict 2PL has the added benefit of avoiding "cascading aborts" (where aborting one transaction forces you to abort others that read its uncommitted data). In practice, when people say "2PL" in a database context, they almost always mean strict 2PL.

4. Why Strict 2PL Works

Theorem: If a schedule follows Strict 2PL, it is conflict serializable (and therefore serializable, giving us both isolation and consistency).

Why does this work? The intuition is that locks force an ordering on conflicting operations. If T1 and T2 both want to access the same object, and at least one of them writes, then one of them will get the lock first and the other must wait. Because strict 2PL holds locks until commit, the waiting transaction cannot proceed until the first one is entirely done with that object (and in fact, entirely done with all objects). This forces the schedule to be "equivalent" to running the transactions one after another.

What Strict 2PL gives us

  • ✓ Every Strict 2PL schedule is conflict serializable
  • ✓ Conflict serializable implies serializable
  • ✓ Serializable implies isolation (each transaction appears to run alone)
  • ✓ Isolation + correct transactions implies consistency

However, there is an important caveat. Strict 2PL produces only a subset of all conflict serializable schedules. There exist schedules that are perfectly conflict serializable but would never be produced by strict 2PL, because 2PL is more conservative than necessary. This is the price we pay for a simple, practical algorithm — we give up some concurrency.

The One Subtle Problem

The theorem says "if the schedule follows strict 2PL, it is conflict serializable." But there is a critical hidden assumption: the schedule must complete. What if two transactions are each waiting for the other to release a lock? Neither can proceed. Neither will ever commit. The schedule never completes. This is a deadlock, and it is the Achilles' heel of locking-based concurrency control.

5. Deadlocks

Deadlock: A situation where a cycle of transactions exists, with each transaction waiting for the next transaction in the cycle to release a lock. No transaction in the cycle can proceed, so all of them wait forever.

Deadlocks are a fundamental problem, not a bug. They arise naturally from the combination of (1) transactions holding locks while waiting, and (2) the possibility that two or more transactions need the same set of objects in different orders. Deadlocks cannot be prevented by simply "being more careful" with the locking protocol — we need explicit strategies to deal with them.

Two cars meet on a narrow one-lane bridge, approaching from opposite sides. Each car has entered the bridge (acquired a lock on part of it) and needs the other car to back up (release its lock) before it can proceed. Neither will back up. This is a deadlock. The only solutions are to either prevent the situation from arising (don't both enter the bridge) or detect it and force one car to back up.

There are two fundamental approaches to dealing with deadlocks:

Prevention

Design the protocol so that deadlocks can never occur. This is conservative and may reduce concurrency, but eliminates the problem entirely.

Detection

Allow deadlocks to occur, but periodically check for them and break them by aborting one of the transactions in the cycle.

6. Deadlock Detection: A Step-by-Step Example

Let us walk through the classic deadlock scenario. Two transactions, T1 and T2, each need to read one object and then write another — but they access the objects in opposite orders.

1 T1 acquires S(A) and reads A

T1 wants to read A, so it requests an S lock on A. No one else holds a lock on A, so the lock is granted immediately.

Lock state: T1 holds S(A). Waits-for graph: empty.

2 T2 acquires S(B) and reads B

T2 wants to read B, so it requests an S lock on B. No one else holds a lock on B, so the lock is granted immediately.

Lock state: T1 holds S(A), T2 holds S(B). Waits-for graph: empty.

3 T2 requests X(A) — must WAIT

T2 wants to write A, so it requests an X lock on A. But T1 holds S(A), and X locks are incompatible with S locks. T2 must wait for T1 to release S(A).

Lock state: T1 holds S(A), T2 holds S(B), T2 waiting for X(A). Waits-for graph: T2 → T1

4 T1 requests X(B) — must WAIT — DEADLOCK!

T1 wants to write B, so it requests an X lock on B. But T2 holds S(B), and X locks are incompatible with S locks. T1 must wait for T2 to release S(B).

Lock state: T1 holds S(A) and waits for X(B), T2 holds S(B) and waits for X(A).

T1 → T2 → T1   =   CYCLE = DEADLOCK

T1 is waiting for T2 (to release S(B)), and T2 is waiting for T1 (to release S(A)). Neither can make progress. Under strict 2PL, locks are only released at commit/abort, and neither transaction can reach commit because it is blocked. This is a deadlock.

Important Distinction

The waits-for graph is different from the conflict graph (precedence graph) we discussed in Part 1. The conflict graph has an edge T_i → T_j if an operation of T_i conflicts with and precedes an operation of T_j — it captures the serialization order. The waits-for graph has an edge T_i → T_j if T_i is currently blocked waiting for T_j to release a lock — it captures the current blocking relationships. A cycle in the conflict graph means the schedule is not conflict serializable. A cycle in the waits-for graph means there is a deadlock.

7. Deadlock Prevention Strategies

Strategy 1: Conservative 2PL (C2PL)

Conservative 2PL (C2PL): Before a transaction begins executing any operations, it must acquire all the locks it will ever need. If it cannot get all of them at once, it releases any locks it has obtained and tries again later.

Why does this prevent deadlocks? Because a transaction never holds some locks while waiting for others. It either has everything it needs (and can run to completion without waiting) or it has nothing (and is not blocking anyone). Deadlock requires a cycle of "holding while waiting," and C2PL eliminates the "holding while waiting" condition entirely.

Imagine a chef who collects all ingredients before starting to cook. If even one ingredient is unavailable, they put everything back and wait. They never stand at the stove holding half the ingredients while blocking another chef from accessing the fridge. No standoffs are possible.

Downsides of C2PL

  • Reduced concurrency: Locks are held for the entire duration of the transaction, starting even before execution begins. Objects are locked longer than necessary.
  • Requires foreknowledge: The system must know in advance which objects the transaction will access. This is often impractical for complex queries where the set of accessed objects depends on data values.
  • Potential livelock: A transaction might repeatedly fail to acquire all locks and keep retrying indefinitely, though this is usually addressed with backoff strategies.

Strategy 2: Wait-Die and Wound-Wait

These two schemes use transaction priorities to decide what happens when a lock request would cause waiting. The priority is based on the transaction's start timestamp: older transactions (started earlier) have higher priority. If T1 started before T2, then T1 has higher priority than T2.

In both schemes, when transaction Tx requests a lock held by Ty, the system compares their priorities and takes action. The key difference is who gets aborted.

Wait-Die (Non-Preemptive)

If Tx has higher priority (is older) than Ty: Tx waits for Ty.

If Tx has lower priority (is younger) than Ty: Tx aborts (dies).

Intuition: Old transactions are patient and wait; young transactions give up easily. The lock holder (Ty) is never forced to abort — hence "non-preemptive."

Wound-Wait (Preemptive)

If Tx has higher priority (is older) than Ty: Ty aborts (is "wounded" by Tx).

If Tx has lower priority (is younger) than Ty: Tx waits for Ty.

Intuition: Old transactions are aggressive and kick out young ones; young transactions respectfully wait. The lock holder (Ty) can be forced to abort — hence "preemptive."

Wait-Die vs. Wound-Wait Comparison

Scenario Wait-Die Wound-Wait
Tx (older/higher priority) requests lock held by Ty (younger) Tx waits Ty aborts (wounded)
Tx (younger/lower priority) requests lock held by Ty (older) Tx aborts (dies) Tx waits
Who can be forced to abort? Only the requester Only the holder (if younger)
Preemptive? No Yes
Who waits? Older waits for younger Younger waits for older

Why do both prevent deadlocks?

In Wait-Die, a transaction only waits for a younger transaction. So in any chain of waiting, priorities decrease: T_old waits for T_young. You can never form a cycle because that would require T_young to wait for T_old, but Wait-Die says younger transactions abort rather than wait for older ones.

In Wound-Wait, a transaction only waits for an older transaction. So in any chain of waiting, priorities increase: T_young waits for T_old. Again, you cannot form a cycle because that would require T_old to wait for T_young, but Wound-Wait says older transactions wound (abort) younger ones rather than wait.

Both schemes impose a consistent direction on the waits-for edges, making cycles impossible.

When an aborted transaction restarts, it keeps its original timestamp. This is critical — it ensures the restarted transaction has high priority and will not keep getting aborted over and over again. Eventually, it will be the oldest transaction and will succeed.

8. Deadlock Detection

Instead of preventing deadlocks, we can allow them to happen and then detect and resolve them. This is the more common approach in practice because it allows greater concurrency — transactions are not prematurely aborted or forced to acquire all locks upfront.

Waits-for Graph: A directed graph where each node represents a transaction and a directed edge Ti → Tj means Ti is waiting for Tj to release a lock. A cycle in this graph indicates a deadlock.

The Detection Algorithm

  1. Maintain the waits-for graph as transactions request and release locks.
  2. Periodically check the graph for cycles (or check whenever a new wait edge is added).
  3. If a cycle is found, select a victim transaction in the cycle and abort it. This breaks the cycle and allows the other transactions to proceed.
  4. The victim is typically chosen to minimize wasted work (e.g., the youngest transaction or the one that has done the least work).

The frequency of deadlock checking is a tuning parameter. Checking too often adds overhead; checking too rarely means transactions sit idle in deadlocks for longer. In practice, many systems check every few seconds or whenever a transaction has been waiting longer than a threshold.

9. Detailed Example: 2PL Execution with Five Transactions

Let us trace through a more complex example to see strict 2PL in action. We have five transactions with the following operations to execute:

Operations:
w1(A),   r2(A),   w1(B),   w3(C),   r2(C),   r4(B),   w2(D),   w4(E),   r5(D),   w5(E)

We process these operations in the order listed. For each operation, we attempt to acquire the necessary lock. If the lock cannot be granted, the transaction waits. When a transaction finishes all its operations (and no more are listed), it commits and releases all locks.

1 w1(A): T1 requests X(A)

No lock exists on A. T1 is granted X(A). T1 writes A.

Locks held: T1: X(A). Waiting: none.

2 r2(A): T2 requests S(A) — BLOCKED

T2 wants to read A, so it requests S(A). But T1 holds X(A), and S is incompatible with X. T2 must wait for T1 to release X(A).

Locks held: T1: X(A). Waiting: T2 waits for T1 (on A). Waits-for graph: T2 → T1.

3 w1(B): T1 requests X(B)

No lock exists on B. T1 is granted X(B). T1 writes B. T1 has now completed all its operations (w1(A) and w1(B)).

Locks held: T1: X(A), X(B). Waiting: T2 waits for T1.

4 T1 COMMITS — releases X(A), X(B)

T1 has completed all its operations. It commits and releases all locks. Now T2's request for S(A) can be granted.

T2 unblocked: T2 is granted S(A) and reads A.

Locks held: T2: S(A). Waiting: none. Waits-for graph: empty.

5 w3(C): T3 requests X(C)

No lock exists on C. T3 is granted X(C). T3 writes C. T3 has completed all its operations.

Locks held: T2: S(A); T3: X(C). Waiting: none.

6 r2(C): T2 requests S(C) — BLOCKED

T2 wants to read C, so it requests S(C). But T3 holds X(C). T2 must wait for T3.

Locks held: T2: S(A); T3: X(C). Waiting: T2 waits for T3 (on C). Waits-for graph: T2 → T3.

7 T3 COMMITS — releases X(C)

T3 has completed all its operations. It commits and releases X(C). T2 is unblocked.

T2 unblocked: T2 is granted S(C) and reads C.

Locks held: T2: S(A), S(C). Waiting: none.

8 r4(B): T4 requests S(B)

No lock exists on B (T1 released it). T4 is granted S(B). T4 reads B.

Locks held: T2: S(A), S(C); T4: S(B). Waiting: none.

9 w2(D): T2 requests X(D)

No lock exists on D. T2 is granted X(D). T2 writes D. T2 has now completed all its operations (r2(A), r2(C), w2(D)).

Locks held: T2: S(A), S(C), X(D); T4: S(B). Waiting: none.

10 T2 COMMITS — releases S(A), S(C), X(D)

T2 has completed all its operations. It commits and releases all its locks.

11 w4(E): T4 requests X(E)

No lock exists on E. T4 is granted X(E). T4 writes E. T4 has completed all its operations (r4(B), w4(E)).

Locks held: T4: S(B), X(E). Waiting: none.

12 T4 COMMITS — releases S(B), X(E)

T4 has completed all its operations. It commits and releases all its locks.

13 r5(D): T5 requests S(D)

No lock exists on D (T2 released it). T5 is granted S(D). T5 reads D.

Locks held: T5: S(D). Waiting: none.

14 w5(E): T5 requests X(E)

No lock exists on E (T4 released it). T5 is granted X(E). T5 writes E. T5 has completed all its operations (r5(D), w5(E)).

Locks held: T5: S(D), X(E). Waiting: none.

15 T5 COMMITS — releases S(D), X(E)

T5 has completed all its operations. It commits and releases all locks. All transactions have completed successfully.

Observations from this execution

  • The waits-for graph at its most complex had just one edge: T2 → T1 (step 2) and later T2 → T3 (step 6). No cycle ever formed, so no deadlock occurred.
  • T2 was blocked twice — first waiting for T1 on object A, then waiting for T3 on object C. Each time, the blocking transaction finished and released its locks, unblocking T2.
  • The final commit order was T1, T3, T2, T4, T5. This schedule is equivalent to the serial schedule T1 → T3 → T2 → T4 → T5.
  • This example completed without deadlock, but that was not guaranteed — a different ordering of operations or different transactions could have produced a cycle in the waits-for graph.
Waits-for graph at Step 2:   T2 → T1
Waits-for graph at Step 6:   T2 → T3
No cycle at any point — no deadlock.

10. Summary

Key Takeaways

  • Strict 2PL is the workhorse concurrency control protocol: acquire S locks before reads, X locks before writes, release everything at commit/abort.
  • If a strict 2PL schedule completes, it is guaranteed to be conflict serializable — giving us isolation and consistency.
  • Strict 2PL produces a subset of all conflict serializable schedules. It is conservative, trading some concurrency for simplicity and correctness.
  • Deadlocks are the critical weakness: cycles in the waits-for graph where transactions are stuck waiting for each other forever.
  • Prevention strategies (C2PL, Wait-Die, Wound-Wait) make deadlocks impossible but may reduce concurrency or cause unnecessary aborts.
  • Detection strategies maintain a waits-for graph, periodically check for cycles, and abort a victim to break the cycle. This is the most common approach in practice.
  • The fundamental limitation of locking: there is no guarantee the schedule will finish. Correctness is conditional on completion.
Locking-based concurrency control gives us a powerful guarantee: every schedule it produces is conflict serializable. But it comes with a price — the possibility of deadlock means we need additional machinery (prevention or detection) to ensure that transactions actually complete. The database does not simply "run transactions correctly" — it runs them correctly if it can get them to finish, and it uses deadlock handling to make sure they eventually do.
