1. ACID — The Four Guarantees

TL;DR: Every database transaction promises four things: it either fully happens or doesn't (Atomicity), it leaves data valid (Consistency), concurrent transactions don't interfere (Isolation), and committed data survives crashes (Durability).

Each DBMS technique maps to specific ACID properties

PropertyWhat it meansHow it's enforced
AtomicityAll or nothing — no partial transactionsWrite-Ahead Logging (WAL)
ConsistencyData stays valid (constraints hold)2-Phase Locking + app logic
IsolationTransactions don't see each other's messSerializability / 2PL
DurabilityCommitted = permanent, even after crashWrite-Ahead Logging (WAL)

2. The Four Concurrency Anomalies

TL;DR: When transactions run concurrently, four bad things can happen. Serializability prevents all of them. Weaker isolation levels allow some.

All four anomalies are prevented by serializability

AnomalyTypeOne-line explanation
Lost UpdateWrite-WriteTwo writes to same row → one silently overwritten
Dirty ReadWrite-ReadReading data from an uncommitted (maybe aborted) transaction
Unrepeatable ReadRead-WriteSame row, two reads, different values (someone changed it in between)
Phantom ReadRead-WriteSame query, two runs, different set of rows (someone inserted/deleted)
Key distinction: Unrepeatable read = a single row changed its value. Phantom read = the set of rows matching a query changed (rows appeared/disappeared).

3. Conflict Serializability

TL;DR: A schedule is "safe" if its result matches some serial (one-at-a-time) execution. Test this by building a conflict graph: no cycle = serializable.

What's a conflict?

Two operations from different transactions on the same data item where at least one is a write. (Two reads are never a conflict.)

Serial schedule (always safe) vs. interleaved schedules (may or may not be safe)

The 3-step test

  1. Find conflicts — scan for RW, WR, WW pairs on same data item between different transactions
  2. Build conflict graph — edge from Ti → Tj if Ti's conflicting op comes first
  3. Check for cycles — acyclic = conflict serializable. Topological sort gives equivalent serial order.

5-transaction example: find conflicts → build graph → acyclic → equivalent serial order T3, T1, T4, T2, T5

4. Two-Phase Locking (2PL)

TL;DR: Before reading, get a Shared (S) lock. Before writing, get an Exclusive (X) lock. Release everything only at commit/abort. This guarantees conflict serializability.

The three rules of Strict 2PL

Lock compatibility (the key table)

Request \ HeldNoneS (Shared)X (Exclusive)
S lockGrantedGrantedWAIT
X lockGrantedWAITWAIT

Readers can coexist (S+S = OK). Writers need total exclusivity (X + anything = WAIT).

Growing phase (acquire locks) → all locks released at commit. Strict 2PL = release everything at the end.

Why it works: Locks force an ordering on conflicting operations. If T1 and T2 conflict, one waits for the other to finish. Since locks are held until commit, the result is always equivalent to some serial order.
The catch: Strict 2PL guarantees correctness if the schedule completes. But it might not complete — because of deadlocks.

5. Deadlocks

TL;DR: Deadlock = cycle of transactions each waiting for the other's lock. Solution: either prevent them or detect and break them.

T1 waits for T2 (holds S(B)), T2 waits for T1 (holds S(A)) → cycle = DEADLOCK

Waits-for graph ≠ Conflict graph! Conflict graph = serialization order (cycle = not serializable). Waits-for graph = who's blocked on whom right now (cycle = deadlock). Different graphs, different purposes.

Three ways to handle deadlocks

Prevention: C2PL

Get all locks before starting. If you can't get all, release everything and retry. No deadlock possible, but hurts performance.

Detection

Let deadlocks happen. Periodically check waits-for graph for cycles. If found, abort one victim to break the cycle. Most common in practice.

Wait-Die vs. Wound-Wait: priority-based deadlock prevention

Wait-Die vs Wound-Wait (cheat sheet)

ScenarioWait-DieWound-Wait
Older requests lock from youngerOlder waitsYounger aborts
Younger requests lock from olderYounger abortsYounger waits
Why no deadlock?Waits only go old→young (no cycle)Waits only go young→old (no cycle)

Key: Aborted transactions keep their original timestamp, so they get higher priority when they retry.

6. Isolation Levels

TL;DR: Full serializability is expensive. Real databases trade correctness for speed. Most run at Read Committed — which still allows lost updates, unrepeatable reads, and phantoms.

Real-world isolation guarantees — serializability is unfortunately rare

LevelDirty ReadLost UpdateUnrepeatable ReadPhantom

Read CommittedPreventedAllowedAllowedAllowed

Strict Serializability

Even stronger than serializability: if transaction X finishes before Y starts, then X must come before Y in the equivalent serial order. 2PL naturally gives you this.

MVCC (Multi-Version Concurrency Control)

The modern approach: Instead of blocking readers with locks, keep multiple versions of data. Readers see a consistent snapshot while writers create new versions. Used by PostgreSQL, MySQL, Oracle, and virtually all modern databases.

7. Indexing

TL;DR: An index is a small lookup table of (key, pointer) pairs that lets you find rows without scanning the entire table. Like a book's index — look up the topic, get the page number.
Index = (search-key, pointer) — typically 100x smaller than the data. Stored in memory when possible for fast lookups. Pointer size depends on data size (e.g., 30 bits for 1GB).

Primary vs Secondary Index

Primary (Clustering)

Index order = physical data order. Only one per table. Can be sparse.

Secondary (Non-clustering)

Index order ≠ data order. Many per table. Must be dense (can't scan sequentially).

Dense vs Sparse Index

Dense index: one entry for every search-key value → direct pointer to each record

Sparse index: one entry per block → find nearest, then scan within block

Dense Index

  • Entry for every key value
  • Fast lookup (direct pointer)
  • More space + maintenance
  • Works on any attribute

Sparse Index

  • Entry for some key values
  • Slower (need to scan within block)
  • Less space + maintenance
  • Only works if data is sorted by key

Secondary Index

Secondary index on salary: points to buckets since data isn't sorted by salary. Must be dense.

Multilevel Index

When the index itself is too big for memory: build an index on top of the index. The outer index is sparse, the inner index is the original. Add more levels if needed (this leads to B+ trees).

Outer index → inner index → data blocks. Each level narrows the search.
