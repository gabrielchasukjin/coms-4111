FIFO and LRU Cache Eviction

Two fundamental eviction policies — simplicity vs. recency awareness

1. FIFO (First In, First Out)

FIFO Policy: Evict the item that was inserted into the cache earliest (i.e., has been in the cache the longest). The idea is simple: the oldest item is the least relevant.

Advantages

  • Simple and easy to implement
  • Can be implemented as a basic queue — new items enter at the tail, evictions happen at the head
  • No need to track access patterns at all

Disadvantages

  • An old item might actually be a popular item that is accessed frequently
  • FIFO ignores usage entirely — it doesn't care how often or how recently an item was accessed
  • In general, suffers from a low hit rate

FIFO Worked Example

Access sequence: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 with a cache of size 4. The leftmost item in the cache is the oldest (next to be evicted).

Request Result Action Cache State (oldest → newest)
1 Miss Insert 1 [1]
2 Miss Insert 2 [1, 2]
3 Miss Insert 3 [1, 2, 3]
4 Miss Insert 4 (cache full) [1, 2, 3, 4]
1 Hit 1 is in cache [1, 2, 3, 4]
2 Hit 2 is in cache [1, 2, 3, 4]
5 Miss Evict 1 (oldest), insert 5 [2, 3, 4, 5]
1 Miss Evict 2, insert 1 [3, 4, 5, 1]
2 Miss Evict 3, insert 2 [4, 5, 1, 2]
3 Miss Evict 4, insert 3 [5, 1, 2, 3]
4 Miss Evict 5, insert 4 [1, 2, 3, 4]
5 Miss Evict 1, insert 5 [2, 3, 4, 5]

Result: 10 misses out of 12 requests

Hit rate: 2/12 = 16.7%. This is terrible! After the initial cold misses (filling the cache with 1-4), items 1 and 2 get hits on requests 5-6. But then requesting 5 evicts 1 (the oldest), and from that point on, every single request is a miss.

The problem is clear: FIFO evicted items 1 and 2 even though they were recently accessed (and about to be accessed again). It only cares about insertion time, not usage.

2. LRU (Least Recently Used)

LRU Policy: Evict the item that was least recently accessed (i.e., whose last access was furthest in the past). The idea: if an item was accessed recently, it is likely to be accessed again soon.

The key difference from FIFO: LRU tracks when each item was last used, not just when it was first inserted. Every time an item is accessed (hit), its "last used" timestamp is refreshed, moving it to the back of the eviction queue.

Advantages

  • The default eviction policy in most systems — works well for many real workloads
  • Captures temporal locality: recently used items stay in cache
  • Popular items that are accessed repeatedly will never be evicted (they keep refreshing their position)

Disadvantages

  • Sounds simple but can be expensive to implement
  • Need to track the last access time of each item
  • Need to efficiently find the item with the oldest access time (often done with a doubly-linked list + hash map)

LRU Worked Example (Beginning)

Same access sequence: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 with cache size 4. The cache is ordered by recency — leftmost is the least recently used (next to be evicted).

First 4 Requests: Cold Misses

Request 1: miss → cache = [1]

Request 2: miss → cache = [1, 2]

Request 3: miss → cache = [1, 2, 3]

Request 4: miss → cache = [1, 2, 3, 4] (full)

So far, LRU behaves identically to FIFO — the cache fills up with 4 misses. The difference begins on the next request, because LRU will update the position of items when they are accessed. The full trace continues in Part 7.

The critical difference is what happens next: when request 1 arrives (a hit), FIFO does nothing to 1's position in the queue. But LRU moves 1 to the "most recently used" end, protecting it from eviction. This means when item 5 later needs space, LRU will evict 3 (the least recently used) instead of 1 (which FIFO would evict).

3. FIFO vs. LRU: The Core Distinction

Property FIFO LRU
Evicts based on Insertion time Last access time
On cache hit Nothing changes Item moves to "most recent" position
Tracks Only insertion order Last access time for every item
Implementation Simple queue Doubly-linked list + hash map
Handles popular items Poorly — evicts them if old Well — recent access protects them
Typical hit rate Low Good for most workloads
Think of a library shelf with limited space. FIFO is like removing the book that was placed on the shelf earliest, regardless of whether students keep borrowing it. LRU removes the book that nobody has checked out for the longest time — a much better heuristic for keeping useful books available.

Key Takeaways from Part 6

  • ✓ FIFO evicts the oldest-inserted item — simple (queue) but ignores usage, leading to low hit rates
  • ✓ FIFO example: 10 misses out of 12 requests (16.7% hit rate) on the reference sequence
  • ✓ LRU evicts the least-recently-accessed item — captures temporal locality and is the default policy in most systems
  • ✓ The key difference: on a cache hit, FIFO does nothing while LRU refreshes the item's position, protecting frequently accessed items from eviction
