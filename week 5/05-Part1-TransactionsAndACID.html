Transactions and OLTP Databases

Understanding how databases keep your data safe and consistent

Why Transactions Matter

Imagine you are transferring $500 from your checking account to your savings account. The database needs to do two things: subtract $500 from checking and add $500 to savings. What happens if the system crashes right after the subtraction but before the addition? Your $500 just vanished into thin air. This is the kind of nightmare that transactions are designed to prevent.

Transaction: A sequence of one or more database operations (reads and writes) that are treated as a single, indivisible unit of work. Either all of the operations succeed and their effects become permanent, or none of them do. There is no in-between.

Transactions are the foundational abstraction of OLTP (Online Transaction Processing) databases — the systems that power nearly every application you interact with daily: banking, e-commerce, social media, ride-sharing, and more. These systems handle enormous volumes of short, fast operations from many users simultaneously, and they must do so without corrupting data or producing nonsensical results.

Think of a transaction like an "all-or-nothing" contract. When you buy a house, the deed transfer, the mortgage signing, and the payment all happen together at closing. If any single piece falls through, the entire deal is off — you do not end up paying for a house you do not own, and the seller does not transfer a house they have not been paid for. A database transaction provides exactly this guarantee for data operations.

The ACID Properties

The guarantees that transactions provide are captured by a famous acronym: ACID. Each letter represents a property that the database management system (DBMS) must enforce. Let us look at each one and understand the specific techniques that make them possible.

ACID stands for Atomicity, Consistency, Isolation, and Durability. Together, these properties ensure that database transactions are processed reliably, even in the face of errors, crashes, and concurrent access.

A — Atomicity

A transaction is "all or nothing." If any part fails, the entire transaction is rolled back and the database is left unchanged.

Enforced by: Write-Ahead Logging (WAL)

C — Consistency

A transaction brings the database from one valid state to another. All integrity constraints (foreign keys, uniqueness, check constraints) are satisfied before and after.

Enforced by: 2-Phase Locking + application logic

I — Isolation

Concurrent transactions do not interfere with each other. Each transaction behaves as if it were the only one running on the system.

Enforced by: Serializability, 2-Phase Locking

D — Durability

Once a transaction commits, its effects are permanent — even if the system crashes immediately afterward.

Enforced by: Write-Ahead Logging (WAL)

The Techniques Behind ACID

A DBMS employs several core techniques and algorithms to deliver these guarantees. Some of these directly enforce ACID properties, while others improve performance so that ACID enforcement does not make the system unacceptably slow.

Write-Ahead Logging (WAL) — Atomicity and Durability

Before any change is written to the actual database pages on disk, a record describing the change is first written to a log file. This log is append-only and written sequentially, making it extremely fast. If the system crashes mid-transaction, the log tells the recovery process exactly which changes were made and whether the transaction committed. Uncommitted changes are undone (atomicity); committed changes are redone if they had not yet reached disk (durability).

The key principle: the log is the source of truth. The actual data pages on disk may be out of date, but the log always tells the full story.

Serializability — Isolation

The gold standard for isolation. A schedule of interleaved transactions is serializable if its outcome is equivalent to some serial execution (one transaction at a time, no interleaving). We will explore this in great detail later in this document.

2-Phase Locking (2PL) — Consistency and Isolation

A protocol where each transaction acquires locks in a "growing phase" (acquiring locks, never releasing) and then releases them in a "shrinking phase" (releasing locks, never acquiring). This guarantees that any interleaved schedule produced under 2PL is conflict-serializable, thus enforcing both consistency and isolation.

Performance Techniques

Enforcing ACID is essential, but it is not free. Locking, logging, and checking serializability all add overhead. The following techniques help databases remain fast despite these costs:

Indexing

Provides faster access to data. Instead of scanning every row, the DBMS uses data structures like B+ trees or hash indexes to jump directly to the needed rows. This is critical for transactions that touch only a few rows out of millions.

Bloom Filters

Reduces unnecessary access. A probabilistic data structure that can quickly tell you "this data is definitely not here" (no false negatives) or "this data might be here" (possible false positives). This avoids expensive disk reads for data that does not exist.

Caching

Reduces work duplication. Frequently accessed data pages are kept in memory (the "buffer pool") so that repeated reads do not require going to disk. The DBMS manages this cache carefully, especially in coordination with write-ahead logging, to ensure that cached dirty pages do not compromise durability.

The ACID properties tell you what guarantees the database provides. The techniques (WAL, 2PL, serializability, indexing, bloom filters, caching) tell you how those guarantees are actually achieved. Understanding both is essential for reasoning about database behavior and performance.

Concurrency Anomalies

When multiple transactions run at the same time and access the same data, things can go wrong in subtle and dangerous ways. These problems are called concurrency anomalies, and understanding them is crucial because they define exactly what the Isolation property of ACID is designed to prevent.

There are four classic anomalies. All of them are disallowed by serializability — if a schedule is serializable, none of these anomalies can occur.

1. Lost Updates (Write-Write Conflict)

The Problem

Two transactions both read the same data item and then both write to it. The second write overwrites the first, and the first transaction's update is silently lost.

Example: A bank account has balance = $1000. Transaction Tx adds $200 (reads $1000, writes $1200). Transaction Ty subtracts $100 (reads $1000, writes $900). If Ty's write happens after Tx's write, the final balance is $900 — the $200 deposit is completely lost. The correct answer should be $1100.

Imagine two people editing the same Google Doc at the same time, but without real-time sync. Person A changes paragraph 3, saves. Person B (who opened the doc before A saved) changes paragraph 5, saves. Person B's save overwrites the entire document, erasing Person A's changes to paragraph 3. Person A's edit is "lost."

2. Dirty Reads (Write-Read Conflict)

The Problem

A transaction reads data that has been modified by another transaction that has not yet committed. If that other transaction later aborts (rolls back), the reading transaction has based its work on data that never officially existed.

Example: Transaction Tx updates account A from $1000 to $500. Transaction Ty reads account A and sees $500. Then Tx aborts, restoring A to $1000. But Ty has already acted on the $500 value — a phantom number that was never committed. Ty has been "dirtied" by reading uncommitted data.

You overhear someone on the phone saying "I'm going to quit my job tomorrow." You immediately tell all your friends. But the person was just venting and never actually quits. You spread information based on an intention that was never finalized — a "dirty read" of someone's uncommitted plans.

3. Unrepeatable Reads (Read-Write Conflict)

The Problem

A transaction reads the same data item twice and gets different values each time, because another transaction modified and committed the data in between the two reads.

Example: Transaction Tx reads A = 0. Transaction Ty then updates A = 1 and commits. Transaction Tx reads A again and now sees A = 1. Within the same transaction, Tx has seen two different values for A. This breaks the illusion that Tx is running in isolation.

You check the price of a flight: $300. You go grab your credit card (takes 30 seconds). You come back and the price is now $450. The "data" you read changed between your two observations, even though from your perspective you were in the middle of a single task (buying a ticket). Unrepeatable reads break the consistency of your view of the world during a transaction.

4. Phantom Reads (Read-Write Conflict)

The Problem

A transaction executes a query that returns a set of rows. Another transaction inserts or deletes rows that match the query's criteria and commits. The first transaction re-executes the same query and gets a different set of rows — "phantom" rows have appeared or disappeared.

Example: Transaction Tx runs SELECT * FROM employees WHERE dept = 'Engineering' and gets 50 rows. Transaction Ty inserts a new engineering employee and commits. Tx runs the same query again and gets 51 rows. A new row — a phantom — has appeared.

You count the number of people in a room: 20. While you are writing down the number, someone walks in. You count again: 21. The room has the same "query" (how many people?) but the result has changed because the underlying set shifted. The new person is a "phantom."
Notice the subtle difference between unrepeatable reads and phantom reads. An unrepeatable read is about a single row changing its value. A phantom read is about the set of rows matching a query changing (rows being added or removed). Both are read-write conflicts, but phantoms involve the boundaries of the result set, not just the content of individual rows.

Summary of Anomalies

Anomaly Conflict Type What Goes Wrong
Lost Update Write-Write (WW) Two transactions write the same item; one write is silently overwritten by the other
Dirty Read Write-Read (WR) A transaction reads data written by another transaction that has not yet committed (and may abort)
Unrepeatable Read Read-Write (RW) A transaction reads the same row twice and gets different values because another transaction modified it in between
Phantom Read Read-Write (RW) A transaction executes the same query twice and gets a different set of rows because another transaction inserted or deleted matching rows

The Unifying Principle

All four anomalies share a common root cause: two transactions access the same data, and at least one of them writes. The combination of concurrent access and modification creates a conflict. Serializability is the guarantee that these conflicts are resolved in a way that is equivalent to running the transactions one at a time — eliminating every anomaly in the table above.

Serializability and Conflict Graphs

We have established that serializability is the gold standard for transaction isolation. But what exactly does it mean, and how do we test whether a given schedule is serializable? This section develops the theory and walks through a complete example.

Serial vs. Interleaved Schedules

Serial Schedule

Transactions execute one at a time, with no interleaving. Transaction T1 runs completely, then T2 runs completely, and so on.

Always correct by definition — no concurrency anomalies are possible. But performance is terrible because there is no parallelism.

Interleaved Schedule

Operations from different transactions are mixed together in time. T1 does a read, then T2 does a write, then T1 does another read, etc.

May or may not be correct. Great for performance (multiple transactions make progress simultaneously), but we need to verify that the result is equivalent to some serial schedule.

Serializable Schedule: An interleaved schedule is serializable if its effect on the database is identical to the effect of some serial schedule of the same transactions. It does not have to match every serial order — just at least one.
Consider a kitchen with multiple chefs. A "serial schedule" is like having each chef complete their entire dish before the next chef starts. An "interleaved schedule" is like chefs sharing the kitchen simultaneously — one chops while another sautés. The schedule is "serializable" if the final result (all dishes) is exactly the same as if the chefs had worked one at a time in some order. The dishes taste the same; only the timing of the kitchen work differs.

Conflict Serializability

Testing whether two schedules produce identical results for every possible database state is impractical. Instead, we use a stronger (more conservative) but easily testable condition called conflict serializability.

Conflicting Operations: Two operations from different transactions conflict if they access the same data item and at least one of them is a write. The three types of conflicts are:
  • Read-Write (RW): One transaction reads, another writes the same item
  • Write-Read (WR): One transaction writes, another reads the same item
  • Write-Write (WW): Both transactions write the same item

Two reads on the same item by different transactions are not a conflict — reads do not change data, so their order does not matter. A conflict exists only when at least one operation is a write, because writes change the state that the other operation depends on.

Conflict Serializable: A schedule is conflict serializable if it can be transformed into a serial schedule by swapping adjacent non-conflicting operations. Equivalently, a schedule is conflict serializable if and only if its conflict graph (also called a precedence graph or serialization graph) is acyclic.

The Conflict Graph Method

The conflict graph gives us a clean, mechanical procedure for testing conflict serializability. Here is the method:

1 Create a node for each transaction

Draw one node per transaction in the schedule. If there are five transactions T1 through T5, you draw five nodes.

2 Identify all conflicts

Scan through the schedule. For every pair of operations from different transactions that access the same data item where at least one is a write, record the conflict. The direction matters: the transaction whose operation comes first in the schedule "precedes" the other.

3 Draw directed edges for each conflict

For each conflict, draw an edge from the transaction that acts first to the transaction that acts second. If Ti's operation on data item X comes before Tj's conflicting operation on X, draw an edge Ti → Tj.

4 Check for cycles

If the conflict graph has no cycles, the schedule is conflict serializable. If it has a cycle, it is not conflict serializable.

5 Find an equivalent serial schedule

If the graph is acyclic, perform a topological sort on the graph. Any valid topological ordering gives you an equivalent serial schedule.

Why does acyclicity imply serializability? Each edge Ti → Tj means "Ti must come before Tj in any equivalent serial schedule." A cycle like Ti → Tj → Ti would mean Ti must come before Tj AND Tj must come before Ti — a logical impossibility. No serial schedule can satisfy contradictory ordering requirements, so a cycle means no equivalent serial schedule exists.

Worked Example: Five Transactions

Let us walk through a complete example with five transactions and the following schedule S1:

Schedule S1:
w1(A), r2(A), w1(B), w3(C), r2(C), r4(B), w2(D), w4(E), r5(D), w5(E)

Notation: ri(X) means "transaction Ti reads data item X" and wi(X) means "transaction Ti writes data item X."

The operations, in order, are:

  1. w1(A) — T1 writes A
  2. r2(A) — T2 reads A
  3. w1(B) — T1 writes B
  4. w3(C) — T3 writes C
  5. r2(C) — T2 reads C
  6. r4(B) — T4 reads B
  7. w2(D) — T2 writes D
  8. w4(E) — T4 writes E
  9. r5(D) — T5 reads D
  10. w5(E) — T5 writes E
1 Identify all conflicts by data item

We examine each data item and find pairs of operations from different transactions where at least one is a write.

Data item A:

  • w1(A) then r2(A) — WR conflict: T1 → T2

Data item B:

  • w1(B) then r4(B) — WR conflict: T1 → T4

Data item C:

  • w3(C) then r2(C) — WR conflict: T3 → T2

Data item D:

  • w2(D) then r5(D) — WR conflict: T2 → T5

Data item E:

  • w4(E) then w5(E) — WW conflict: T4 → T5
2 Build the conflict graph

We have five nodes (T1, T2, T3, T4, T5) and the following directed edges:

Edge Reason Data Item
T1 → T2 w1(A) before r2(A) — WR A
T1 → T4 w1(B) before r4(B) — WR B
T3 → T2 w3(C) before r2(C) — WR C
T2 → T5 w2(D) before r5(D) — WR D
T4 → T5 w4(E) before w5(E) — WW E

The conflict graph looks like this:

T3 → T2
T1 → T2
T1 → T4
T2 → T5
T4 → T5

Visually: T3 points to T2. T1 points to both T2 and T4. T2 and T4 both point to T5. The graph flows from left to right: {T3, T1} → {T2, T4} → {T5}.

3 Check for cycles

Let us trace all possible paths through the graph:

  • T3 → T2 → T5 (dead end)
  • T1 → T2 → T5 (dead end)
  • T1 → T4 → T5 (dead end)

No path leads back to a node it has already visited. The graph is acyclic. Therefore, schedule S1 is conflict serializable.

4 Find an equivalent serial schedule via topological sort

A topological sort orders the nodes so that for every directed edge Ti → Tj, Ti appears before Tj. We process nodes with no incoming edges first:

  1. T3 has no incoming edges. Pick T3 first. Remove T3 and its outgoing edges.
  2. T1 has no incoming edges. Pick T1 next. Remove T1 and its outgoing edges.
  3. T4 now has no incoming edges (its only incoming edge was from T1, which is removed). Pick T4. Remove T4 and its edges.
  4. T2 now has no incoming edges (its incoming edges from T1 and T3 are removed). Pick T2. Remove T2 and its edges.
  5. T5 is the only remaining node. Pick T5.

Equivalent Serial Schedule

T3, T1, T4, T2, T5

This means: if we ran T3 first (completing entirely), then T1, then T4, then T2, then T5 — all one at a time with no interleaving — the final database state would be identical to what schedule S1 produces. The interleaved schedule S1 is therefore safe: it achieves the same result as this serial order, but with the performance benefit of concurrent execution.

Note that the topological sort is not necessarily unique. For instance, T1, T3, T2, T4, T5 is also a valid topological ordering (you can verify that all edges still go left to right). Both are valid equivalent serial schedules.

The conflict graph method is powerful because it is entirely mechanical. You do not need to reason about what the data values are, what the transactions "mean," or simulate the entire execution. You just find conflicts, draw edges, and check for cycles. If acyclic, the schedule is safe. If cyclic, it is not, and the DBMS must prevent or resolve the situation (typically by aborting one of the transactions involved in the cycle).

Putting It All Together

Let us step back and see how the concepts we have covered form a coherent picture of how OLTP databases work.

Key Takeaways

  • ✓ Transactions are the fundamental unit of work in OLTP databases, providing all-or-nothing guarantees through the ACID properties.
  • ✓ ACID is enforced by concrete mechanisms: Write-Ahead Logging handles Atomicity and Durability; Serializability and 2-Phase Locking handle Isolation and Consistency.
  • ✓ Performance techniques like indexing, bloom filters, and caching ensure that ACID enforcement does not cripple throughput.
  • ✓ Concurrency anomalies — lost updates, dirty reads, unrepeatable reads, and phantom reads — are the specific dangers of running transactions concurrently. All are prevented by serializability.
  • ✓ Conflict serializability is tested using conflict graphs: identify conflicts between transactions, draw directed edges, and check for cycles. An acyclic graph means the schedule is safe; a topological sort of the graph gives an equivalent serial order.
  • ✓ Conflicts arise whenever two different transactions access the same data item and at least one of them writes (RW, WR, or WW). Two reads are never a conflict.
